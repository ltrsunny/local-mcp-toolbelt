## Decisions

### D1 — Backend strategy: **2-pronged for v0.5.0** (revised 2026-05-06, was 3-pronged)

After round-3 the 8-voice meeting converged on a 3-pronged hedge. HEAD-verify (2026-05-06) eliminated Prong C (see D7). Effective strategy:

| Path | Role | Stack |
|---|---|---|
| **MLX (primary)** | Tier-D production | Python/FastAPI HTTP bridge wrapping `mlx-lm`, called from existing `LlmBackend` interface |
| **llama.cpp / Mistral-Nemo-12B** | Fallback + Linux-future-proof | unchanged `node-llama-cpp` v3 backend |
| ~~llama.cpp speculative-decoding spike~~ | ~~dropped from v0.5.0~~ | blocked — see D7 |

**Justification**:
- **Gem (round 2)** + **mistral-large-3 (round 3 turn 3)**: GGUF/Metal cannot get a 14B under 60s on this hardware. Mistral's benchmark estimate: Nemo-12B GGUF=78s, Qwen3-14B-MLX=44s.
- **DeepSeek-V4-Pro (turn 6)**: Mistral's 44s number lacks methodology disclosure; real end-to-end with FastAPI HTTP framing is likely 55-62s. Margin is thin.
- **DeepSeek + Kimi-K2.6 (turn 7)**: speculative-decoding hybrid (Qwen3-1.8B draft + Qwen3-8B verifier, 6.2 GB total, same tokenizer/architecture) could 2-3× decode on llama.cpp without any backend swap. Unproven but architecturally elegant.
- **Llama-3.1-405B (turn 8)**: 3-pronged is "calculated sprawl" justified by zero headroom in the 60s wall — if any single path fails benchmarks, project still ships.

### D2 — Tier-D primary candidate: Qwen3-14B-MLX-4bit

- mlx-community / Qwen3-14B-4bit (verify availability + size before download per D6)
- Apache-2.0 license inherited from upstream Qwen3
- Run via FastAPI HTTP bridge (see D3)

### D3 — MLX integration: FastAPI HTTP bridge (NOT Python subprocess, NOT native bindings)

- **Reject Python subprocess**: cold-start latency unacceptable for per-call MCP tool invocation; brittle stdin/stdout JSON-RPC framing for streaming tokens
- **Reject native MLX bindings from Node**: experimental, segfault risk in MCP host process
- **Accept FastAPI HTTP bridge**: isolates Python crashes, OpenAI-compatible endpoint, lifecycle managed independently, lets us iterate on MLX without recompiling Node binary

`LlmBackend` adds a third implementation `MlxHttpBackend` calling `localhost:<port>/v1/chat/completions`. `backendForTool()` resolves Tier-D to MLX-bridge by default with a fallback chain.

### D4 — License policy: Apache-2.0 / MIT / BSD only (unchanged from Draft 1)

5/5 round-1 reviewers + 8/8 round-3 voices unanimous. Reject:
- `Jackrong/Qwen3.5-9B-DeepSeek-V4-Flash-GGUF` (frankenmerge, claimed Apache but no upstream)
- `RichardErkhov/saiga_gemma_9b-gguf` (no license file, Gemma upstream is restrictive)
- Any community conversion without verifiable LICENSE matching upstream's permissive terms

Llama-license / Gemma-license: gray-listed (case-by-case, not default).

### D5 — Validation requirements (NEW — added by DeepSeek + Kimi)

**Mandatory before promoting Qwen3-14B-MLX to Tier D**:
- End-to-end wall-clock measurement via `time curl` against the running FastAPI server (NOT internal mlx-lm decoder timers — they exclude HTTP framing overhead)
- 5-trial benchmark per task; report median + 95th percentile
- 95th percentile must be **<55 s** on tasks 01/03/04/05 (5 s margin under the 60 s MCP wall)
- Peak RAM (sampled via macOS `ps -p PID -o rss=` per existing harness) <12 GB

If 95th percentile is >55 s, DO NOT promote — fall back to Mistral-Nemo-12B on llama.cpp (rank 2) or trigger spec-decoding spike (D7).

### D6 — Pre-eval verification step (mandatory; carries over from Draft 1)

Before downloading any candidate run a HEAD-request sweep (use copilot `--allow-all-paths` to grant /tmp + project access):
- Confirm official upstream HF repo (e.g. `mlx-community/Qwen3-14B-4bit`)
- Confirm Content-Length and that LICENSE file matches upstream's permissive terms
- Reject any artifact >11 GB or with missing/restrictive LICENSE

### D7 — Speculative-decoding spike: **DROPPED from v0.5.0** (2026-05-06 HEAD-verify)

Double-blocked; cannot proceed in v0.5.0 cycle:

1. **Qwen3 GGUF models not available**: `bartowski/Qwen3-8B-GGUF` and `bartowski/Qwen3-1.8B-GGUF` repos exist but contain **zero GGUF files** (converted models not yet uploaded). Qwen3 official org repo is also empty of GGUF format.
2. **node-llama-cpp v3.18.1 has no speculative decoding API**: Searched changelog, README, and 10 most recent GitHub releases — no "speculative", "draft_model", or "n_draft" found.

**Consequence**: Strategy simplified from 3-pronged to **2-pronged**:
- Path A: Qwen3-14B-MLX-4bit via FastAPI HTTP bridge [primary]
- Path B: Mistral-Nemo-12B Q4_K_M via existing node-llama-cpp [fallback]

Revisit spec-decoding when (a) Qwen3 GGUFs land on bartowski/HF AND (b) node-llama-cpp adds spec-decode API (track upstream).

### Eval run: 20260507092110-mlx-8000 (tasks 03, 04, 05)

| Task | Prompt tok | Completion tok | Latency | tps | Result |
|---|---|---|---|---|---|
| 03-classify | 411 | 855 | 35.3s | 24.2 | ✅ within 60s |
| 04-extract | 1242 | **1962** | **87.6s** | 22.4 | ❌ **exceeds 60s** |
| 05-transform | 1014 | 1098 | 52.9s | 20.8 | ✅ within 60s |

### Root cause analysis

**04-extract failure**: `response_format: { type: 'json_object' }` ensures valid JSON but does NOT enforce the JSON Schema. Qwen3-8B-4bit ignored the schema, included extra fields (abstract, introduction, methodology), and outputted 1962 tokens instead of the expected ~100-200. The latency is entirely output-length driven.

**03-classify quality issue**: Model produced `{"categories": [...]}` instead of `{"labels": [...]}` per schema. Verbose (855 tok vs expected ~50). Same root cause: no grammar constraint.

### Conclusion for v0.5.0

| Tool type | MLX (oMLX) path | Note |
|---|---|---|
| `summarize` / `transform` | ✅ OK | Free-form text, no schema enforcement needed |
| `classify` / `extract` | ❌ Not ready | json_object ≠ schema enforcement; verbose; latency violation |

**Do not promote classify/extract to Tier D** until one of:
1. oMLX adds `json_schema` Structured Outputs support (OpenAI spec §Structured Outputs), OR
2. GBNF-equivalent grammar constraint via mlx-lm `--kv-bits` / outlines integration

Qwen3-8B-4bit throughput (~22-24 tok/s) confirms the latency math: at 22 tok/s, staying under 55s requires ≤1210 completion tokens. Classify/extract tasks must be capped accordingly.

### Task 01 (summarize-long) result

| Task | Prompt tok | Completion tok | Latency | tps |
|---|---|---|---|---|
| 01-summarize-long | 2799 | 2045 | 132.3s | 15.5 |

**Conclusion: Qwen3-8B-4bit cannot serve summarize-long within 60s** (132s = 2.4× limit).

### 8B baseline interpretation

Qwen3-8B-4bit is NOT the intended Tier-D model. The scope memo targets **Qwen3-14B-MLX-4bit** (MLX delivers 30-50% better decode vs llama.cpp for 14B+). The 8B eval serves as an infra sanity check only — confirms oMLX end-to-end stack is working.

At 15-22 tok/s (8B), staying under 55s requires ≤825-1210 completion tokens.
At the projected 14B MLX rate (~20-33 tok/s), the margin improves but long-output tasks remain at risk.

**Next step**: Download `mlx-community/Qwen3-14B-4bit` (~8.3 GB) and re-run eval matrix.

### Qwen3-14B-MLX-4bit eval (2026-05-07)

Run: `20260507094102-mlx-8000`, 1 trial per task.

| Task | Prompt tok | Completion tok | Latency | tps | Status |
|---|---|---|---|---|---|
| 01-summarize-long | 2799 | 580 | 85.2s | 6.8 | ❌ over 60s |
| 03-classify | 411 | 666 | 49.1s | 13.6 | ✅ |
| 04-extract | 1242 | 1456 | 116.1s | 12.5 | ❌ over 60s (schema violation) |
| 05-transform | 1014 | 669 | 58.8s | 11.4 | ✅ (barely) |

**14B decode rate: 6.8–13.6 tok/s** — significantly slower than 8B (15–24 tok/s) on this 16 GB Mac due to memory pressure (14B weights + KV cache fill available VRAM/RAM).

The "30-50% speedup" over llama.cpp is a relative comparison (14B MLX vs 14B GGUF on llama.cpp), not absolute vs 8B. 14B GGUF on llama.cpp would likely be ≤8 tok/s under memory pressure, making 14B MLX still better for 14B — but still slower than 8B-class.

### Root cause: output length, not decode speed

| Task | Expected completion | Actual | Why |
|---|---|---|---|
| 01-summarize-long | 200-400 tok | 580 tok | Long doc → verbose summary |
| 03-classify | 50-100 tok | 666 tok | `explain: true` + no output cap |
| 04-extract | 100-200 tok | 1456 tok | Schema ignored (json_object ≠ grammar) |
| 05-transform | 300-500 tok | 669 tok | Acceptable |

Both failures are **fixable via maxOutputTokens per-task caps**, not requiring a faster model:
- At 12 tok/s, 400-token cap → 33s ✅
- At 7 tok/s (long-prompt TTFT overhead), 400-token cap → 57s ✅

### Tier D promotion decision

| Tool | Verdict | Condition |
|---|---|---|
| `transform` | ✅ promote to Tier D | maxOutputTokens ≤ 600 |
| `classify` | ⚠️ promote with cap | maxOutputTokens ≤ 400; json_object mode is best-effort for schema |
| `summarize-long` | ❌ defer | Needs output cap + quality validation |
| `extract` | ❌ defer | Needs schema enforcement (grammar or json_schema Structured Outputs) |

**v0.5.0 Tier D primary: Qwen3-14B-MLX-4bit via oMLX for transform + classify.**
Summarize and extract remain Tier B/C (llama.cpp GGUF) until output caps + schema enforcement are validated.
