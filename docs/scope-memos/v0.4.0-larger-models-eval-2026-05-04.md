## Problem

Current bridge tier ceiling is `qwen2.5:7b` (Tier C). Real-world frontier-token
savings during v0.2.0 were < 5% (per CLAUDE.md). Root cause: Tier C cannot handle
*hard* tasks — longer summarization, complex extraction, multi-step transforms —
so Claude does them itself.

Hypothesis: a larger local model (Tier D) raises the ceiling enough to capture
significantly more frontier work, even at higher latency.

---

## Hardware constraint

16 GB unified memory Mac (per CLAUDE.md). Current Tier C (`qwen2.5:7b` Q4_K_M
+ 32K context) consumes ~6.7 GB. That leaves ~7–8 GB headroom for OS, browser,
Claude Code, MCP processes.

| Model class | RAM @ 4-bit + 8K ctx | RAM @ 4-bit + 32K ctx | Fits? |
|---|---|---|---|
| 7B | ~5 GB | ~6.5 GB | ✅ (current Tier C) |
| 8B | ~5.5 GB | ~7 GB | ✅ |
| 14B | ~9 GB | ~11 GB | tight, OOM risk at 32K |
| 32B | ~19 GB | OOM | ❌ |
| 70B | ~40 GB | OOM | ❌ |

Tier D ceiling: **~14B at reduced context, or 8B at full context**.

## Eval methodology

Build a hard-task fixture set first, then test all candidates against it.

### Hard-task fixtures (build before any model run)

5 tasks across the 4 tool types, picked from real frontier-Claude work:

1. **summarize-long**: 20 K-word research paper → 200-word abstract preserving 5 key claims
2. **summarize-long-chunked**: 60 K-token podcast transcript → structured outline
3. **classify**: ambiguous tech-vs-policy text with 6-label taxonomy
4. **extract**: nested research-paper metadata (authors, citations, methodology) — flat
   schema only since GBNF subset constraint
5. **transform**: convert messy meeting notes → action-item list with owners

For each fixture: gold reference written by Claude (Opus). Eval = quality vs gold.

### Metrics per (model × task)

- **Quality**: 1–5 score from a Claude judge pass (Sonnet 4.6 as judge, with
  the gold reference as anchor)
- **Latency**: wall-clock seconds end-to-end
- **Memory peak**: resident bytes during call (via `vm_stat` snapshots)
- **Token efficiency**: input + output tokens used

### Promotion criteria for Tier D

A candidate becomes Tier D iff it meets ALL of:
- Quality ≥ 4.0/5.0 average across the 5 fixtures
- Latency < 60 s for tasks 1, 3, 4, 5 (within MCP wall-clock budget)
- Peak RAM < 12 GB (leaves 4 GB headroom)
- Strict equality on at least 3 schema-constrained extractions

---

## Out of scope

- Cloud models (defeats privacy/local-first goal)
- 32B+ models (16 GB hardware can't run them)
- Quality < Q4_K_M quantization (degrades too much)
- New MCP tools (this memo is purely about replacing existing tier ceiling)
