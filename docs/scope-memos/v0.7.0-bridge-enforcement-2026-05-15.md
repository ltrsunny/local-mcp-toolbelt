# v0.7.0 — Bridge enforcement as a shipped product feature

**Status: CANDIDATE (Draft 0, outline only).** Not yet PA'd, not yet
Auditor-passed. This memo records the *direction* extracted from a
real-session pain point during v0.6.0 Day 1-3, so the constraint
doesn't get forgotten.

**Date raised**: 2026-05-15
**Raised by**: user, in reply to Claude's local-only hook prototype —
> 注意强制调用是这个产品最终应该有的特性，而不只是本工程的

## 1. Problem

The "use the bridge for >1KB external content" rule has been in `CLAUDE.md`
since v0.2.0. Real adherence has been < 5%. Soft rules do not survive
contact with a multi-step session. The v0.6.0 cycle confirmed this:
even with explicit memory rules + repeated user reminders + an Auditor
protocol, the assistant kept reading 12KB review outputs whole.

The eventual fix that worked (within this repo) was a PreToolUse hook
that *physically blocks* large reads outside an allow-list, forcing the
assistant to route through the bridge.

That hook currently lives at `.claude/hooks/enforce-bridge.sh` in this
repo only. Any other team that installs `local-mcp-toolbelt` gets the
bridge tools but not the enforcement — exactly the same drift we just
experienced.

**Premise:** enforcement is a *first-class product feature* of the
toolbelt, not an `ollama-claude` private artifact.

### 1.1 Why blocking *file reads* is the right enforcement layer

A natural counter-proposal surfaced on 2026-05-15: "frontier writes
the prompt, local model executes" — make *that* the mandatory pattern
instead of blocking large reads. The token math does not survive
contact with real tasks. Three sub-patterns emerged from the v0.6.0
Day 1-3 retro:

| Sub-pattern | What flows | Bridge saves tokens? | Evidence (this session) |
|---|---|---|---|
| **A. Frontier never touches raw bytes; bridge does.** Large input → small structured output via `source_uri`. | bridge reads N KB → returns < 1 KB struct | **Yes, big** | `extract` on `helpers.sh` (4.8 KB) → two fn defs, footer reports ~600 tok saved. Retro `extract` on merged 3.4 KB drafts: ~505 tok saved. |
| **B. Frontier must understand raw data to write the prompt.** | frontier reads raw + writes prompt + reads model output | **No — net worse** unless the prompt is reused N times. | The #1106 brief: frontier paid the output cost of writing the brief (~500 tok). Only break-even because of fan-out × 3 (copilot + gem + retro `extract`). |
| **C. Task needs judgment, cross-session knowledge, or tone nuance.** | frontier spec → 4B/8B execute → frontier reviews | **No, often worse** — and 4B/8B fail the quality bar anyway. | Retro `extract` on synthesis fan-in caught **zero of three** slips a human caught: a fabricated "MLX RSS ~3 GB" un-observed claim, a relational closing line worth keeping, and an over-prescribed collapse of three discrete brief questions into one baked proposal. See `c9f8a57`. |

The token win is **not** "outsourcing execution to a local model"; it
is "keeping raw bytes out of frontier context", which happens
exclusively in sub-pattern A.

This determines the enforcement layer:

- **Hook-block file reads above threshold** → forces sub-pattern A
  routing automatically. The hook can decide cleanly from the tool
  input alone — it knows the path and the size.
- **Discipline (not hook) for fan-in synthesis and judgment-heavy
  tasks** → sub-pattern B/C calls are semantic state, not file paths.
  No PreToolUse hook can decide them from a `Read` / `Bash` payload.
  Attempting to enforce there would either over-block legitimate work
  or under-block the actual misses.

A v0.7+ shipped product feature inherits this division: `omcp
install-hooks` ships the file-read gate; it does **not** try to
strong-arm sub-pattern B/C tasks into the bridge. Discipline + a
known cost/benefit anchor (the c9f8a57 retro found ~500 tok / ~13 s
per fan-in `extract` is the typical sub-pattern A retrofit cost on
sub-pattern B/C tasks) is the correct layer for B/C.

### 1.2 oMLX cache vs frontier token saving — orthogonal axes

A failure mode that risks confusing both v0.7+ product copy and PA
reviewers: oMLX's hot cache and SSD cache *sound* like they save
tokens. They do not.

Empirical layer-by-layer check from the 2026-05-15 dogfood session:

| Layer | Stores | Saves |
|---|---|---|
| oMLX hot cache (RAM, 6 GB) | model weights + active KV | model cold-load latency |
| oMLX SSD cache (disk, 8.8 GB) | content-addressed KV / weights, 37 MB chunks under `~/.omlx/cache/{0-f}/` | prefill compute on prefix match |
| Bridge tool itself | nothing (it's just an MCP entry point) | **frontier tokens** — and only when sub-pattern A applies (`source_uri`, raw bytes never enter frontier context) |

Concrete numbers from the same session:

- 2 bridge `extract` calls. Wall times 24.9 s and 13.5 s. The ~11 s
  delta on call 2 is consistent with hot-cache warmth post-OOM
  recovery (oMLX restarted at 11:43; first extract paid the model
  cold-load).
- Footer-reported frontier saving: ~1100 tok total (sum of both
  calls' `saved~=+N`). Came entirely from `source_uri` keeping
  `helpers.sh` and the merged-drafts file out of frontier prefill.
  **Cache hit/miss did not change that number.**
- SSD cache mtime audit: zero `.safetensors` written today. Novel
  inputs did not match existing prefixes, but they also did not
  affect frontier billing one way or the other.

The "10-15 % frontier saving share" observed in today's session is
bounded by **sub-pattern A trigger frequency**, not cache hit rate.
Today was process / design heavy (commits, scope memo edits, debate
synthesis = sub-pattern B/C); sub-pattern A only fired twice. An
implementation-heavy or data-ingestion-heavy day would push that
share up sharply.

**Implication for v0.7+ docs and PA review**:

- Cache benefits go to *task throughput / wall-time*, on the local
  machine, where there is no billing.
- Hook enforcement (gating sub-pattern A entry) is the only mechanism
  that affects *frontier billing*.
- Market and document the two separately. Don't let "8.8 GB cache!"
  get conflated with the frontier-saving story.
- PA candidates surveying "local-MCP server enforcement" should be
  filtered for *file-read gating + sub-pattern A routing*, not for
  *local caching* — the latter is table stakes, the former is the
  actual product position.

## 2. Candidate scope (subject to PA + Auditor)

In v0.7.0 (or whichever release), ship:

- **`hooks/`** directory in the published npm package containing
  `enforce-bridge.sh` (generic, env-var-driven — see current prototype).
- **`omcp install-hooks`** CLI subcommand that:
  - Detects the host (Claude Code vs Cursor vs Cline vs Zed vs …) and
    writes the appropriate config (`.claude/settings.json` for Claude
    Code; equivalent extension points for others where they exist).
  - Is **idempotent** — re-running merges, never replaces existing
    hooks.
  - Is **reversible** — `omcp uninstall-hooks` removes only what it
    installed (tag entries with a sentinel).
- **`hooks/README.md`** documenting the env-var knobs:
  `OMCP_HOOK_THRESHOLD_BYTES`, `OMCP_HOOK_ANALYSIS_THRESHOLD_BYTES`,
  `OMCP_HOOK_ANALYSIS_PATHS`, `OMCP_HOOK_DATA_EXTENSIONS`,
  `OMCP_HOOK_EXTRA_ALLOWED_PREFIXES`.

## 3. Out-of-scope (for this proposal)

- Host-side enforcement for clients that have *no* hook mechanism
  (some MCP clients). Possibly addressed by an in-server enforcement
  signal in a later release once SEP-2663 Tasks stabilises.
- Per-session adaptive thresholds. Static env vars are sufficient.
- Telemetry / metric collection. Not selling a service; this is local.

## 4. Open questions for PA + Auditor

1. Is the Claude-Code-only hook surface enough for v0.7.0, or do we
   need Cursor / Cline support on day one? (PA needs a survey of which
   hosts ship a comparable PreToolUse mechanism in 2026.)
2. POSIX-sh-only vs Node implementation of the hook? POSIX-sh is one
   less dep but harder to test; Node ships with the package anyway.
3. Should the install command also be available as a one-shot
   `npx local-mcp-toolbelt install-hooks` for users who don't add the
   server to PATH?
4. What's the upgrade path if a user has customised their hook? We
   should not silently overwrite; the sentinel should let us detect
   "this hook is ours, original" vs "this hook has been modified".
5. SEP-2663 will eventually add server-side capability negotiation —
   does the enforcement model collapse into that, or stay client-side?

## 6. Why not just "tell users to add the hook themselves"

Three reasons:
1. The whole reason the in-repo hook was needed is that *the rule alone
   doesn't work*. The same logic applies to other users of toolbelt —
   asking them to read docs and copy a shell script is the same
   honour-system failure mode at the install layer.
2. Each host has its own hook config schema. Users shouldn't have to
   learn it; the install command handles the impedance.
3. Updates: when the hook gets a CVE fix or a new band, users on
   honour-system installs stay broken. Shipped + versioned is the only
   way to fix that.
