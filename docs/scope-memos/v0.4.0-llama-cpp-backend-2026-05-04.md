**Date**: 2026-05-04  
**Author**: Claude (PM); Opus 4.7 final pass  
**Prior Art**: `docs/prior-art/llama-cpp-backend-2026-05-04.md`  
**Status**: Decisions made; ready to implement

---

## Problem

`ollama-mcp-bridge` runtime-depends on Ollama (daemon + `ollama` npm package).
Ollama has a documented pattern of MIT-license non-compliance, attribution omission,
post-fork performance regression (1.8× slower than llama.cpp), and cloud-model
introduction without disclosure. Full removal is a hard ethical requirement.
Reference: https://sleepingrobots.com/dreams/stop-using-ollama/

---

## Decision

Replace `OllamaBackend` with `LlamaCppBackend` using **`node-llama-cpp` v3**.

**Rationale over `llama-server`**:
- Removes the daemon entirely — users run no background process
- In-process inference: lower latency, simpler install (`npm install`)
- AbortSignal supported natively (`session.prompt({ signal })`)
- Pre-built Metal binaries for macOS arm64 — no compilation required in happy path
- Binary size (~100–200 MB added) is acceptable: project is git-only, no npm publish

---

## Scope

### In

- New `src/llm/llama-cpp-backend.ts` implementing `LlmBackend` interface
- Pre-create one `LlamaContext` per tier at backend construction:
  - Tier B: `contextSize: 4096`
  - Tier C: `contextSize: 32768`
- Grammar: use `llama.createGrammarForJsonSchema()` for `classify` / `extract` tools
  (verify existing schemas are within supported subset — flat objects, expected ✅)
- `countTokens`: `model.tokenize(text).length`
- `ping`: attempt model load / check disposed state
- Remove `ollama` from `dependencies` in `package.json`
- Update `backendForTool()` to instantiate `LlamaCppBackend`
- Update smoke tests (`tests/smoke-bridge.mjs`) for new runtime
- Update README: replace Ollama install instructions with `node-llama-cpp` model download

### Out

- No new MCP tools
- No changes to `LlmBackend` interface contract
- No `llama-server` path (rejected — keeps daemon)
- No llamafile (wrong architecture)
- No multi-model support beyond current tier system
- No streaming exposed to MCP clients (already unused)

## Interface mapping

| Current (`OllamaBackend`) | New (`LlamaCppBackend`) |
|---|---|
| `client.chat({ model, messages, options: { num_ctx } })` | `context.getSequence()` → `session.prompt(text, { signal })` |
| `client.chat({ format: schema })` | `llama.createGrammarForJsonSchema(schema)` → `session.prompt(text, { grammar })` |
| `client.tokenize(text)` | `model.tokenize(text).length` |
| `/api/version` ping | `model.disposed` check |
| Model name from config | GGUF file path from config |

---

## Config change

Current config has `ollamaModel: "qwen3:4b"` etc. (Ollama model tags).
New config needs `modelPath: "/path/to/model.gguf"` per tier.

Propose: new `llamaCppModelPath` field alongside existing fields, with
`ollamaModel` deprecated but not removed (backward compat for anyone still
using Ollama directly). `backendForTool()` selects backend based on which
field is present.

## Risk register

| Risk | Likelihood | Mitigation |
|---|---|---|
| Grammar subset too narrow for existing schemas | Low | Audit 3 schema samples before merge |
| Pre-built binary not available → cmake fallback → slow CI | Medium | Pin a known good version; cache in CI |
| `contextSize` per-tier too coarse (Tier B 4K too small) | Low | Configurable via env var `OMCP_TIER_B_CTX` |
| `node-llama-cpp` API churn (v3 → v4) | Medium | Lock to `^3.18.0` in package.json |

## Decisions (closed)

PM-level, no further user approval needed. Recorded for future sessions.

1. **Binary size (~100–200 MB Metal binaries)**: ✅ accept.
   Project is git-only (no npm publish), so no distribution cost. Users already
   download GGUF model files (multi-GB); incremental binary footprint is negligible.

2. **Config migration**: ✅ accept dual-field.
   New `llamaCppModelPath: "/path/to/model.gguf"` per tier, deprecated
   `ollamaModel` field kept for one minor version. `backendForTool()` selects
   backend by which field is present. Removal in v0.5.0.

3. **Grammar subset**: ✅ accept with mitigation.
   All current `extract`/`classify` schemas are flat objects. Pre-implementation
   step: write a one-shot test that runs `createGrammarForJsonSchema()` against
   each existing schema fixture; if any fail, narrow that schema or post-process.
