## 1. Problem

`local-mcp-toolbelt` v0.6.0 ships as an npm package, but actually
*using* it requires the user to manually:

1. `brew tap jundot/omlx && brew install omlx`
2. `brew services start jundot/omlx/omlx`
3. `npm install local-mcp-toolbelt`
4. `npm run download-models` (~7.5 GB)
5. Edit Claude Desktop / Cursor / Cline config JSON
6. (For full token-saving discipline:) read CLAUDE.md, copy the
   bridge-enforcement hook script, wire it into
   `.claude/settings.json`

Each step is a documented manual operation. The PM's framing on
2026-05-18: "Setup 里面的步骤应该是安装时自动完成，而不是让人自己
操作".

A first-class product fixes this. v0.7+ ships `omcp install` as the
one-shot entrypoint, with sub-commands for partial install paths
(`--models`, `--config`, `--hooks`). The README collapses from "do
3-6 things manually" to `npx local-mcp-toolbelt install`.

The bridge-enforcement hook (originally the v0.7+ scope on
2026-05-15) is now one sub-feature of this broader install — the
same `omcp install` command can `--hooks` into Claude Code's
`settings.json` after writing the MCP server config there.

### 1.1 Why blocking *file reads* is the right enforcement layer

A natural counter-proposal surfaced on 2026-05-15: "frontier writes
the prompt, local model executes" — make *that* the mandatory pattern
instead of blocking large reads. The token math does not survive
contact with real tasks. Three sub-patterns emerged from the v0.6.0
Day 1-3 retro:

| Sub-pattern | What flows | Bridge saves tokens? | Evidence |
|---|---|---|---|
| **A. Frontier never touches raw bytes; bridge does.** Large input → small structured output via `source_uri`. | bridge reads N KB → returns < 1 KB struct | **Yes, big** | `extract` on `helpers.sh` (4.8 KB) → two fn defs, footer reports ~600 tok saved. Retro `extract` on merged 3.4 KB drafts: ~505 tok saved. |
| **B. Frontier must understand raw data to write the prompt.** | frontier reads raw + writes prompt + reads model output | **No — net worse** unless the prompt is reused N times. | The #1106 brief: frontier paid the output cost of writing the brief (~500 tok). Only break-even because of fan-out × 3 (copilot + gem + retro `extract`). |
| **C. Task needs judgment, cross-session knowledge, or tone nuance.** | frontier spec → 4B/8B execute → frontier reviews | **No, often worse** — and 4B/8B fail the quality bar anyway. | Retro `extract` on synthesis fan-in caught **zero of three** slips a human caught. See `c9f8a57`. |

The token win is **not** "outsourcing execution to a local model"; it
is "keeping raw bytes out of frontier context", which happens
exclusively in sub-pattern A.

This determines the hook enforcement layer (part of `omcp install
--hooks`):

- **Hook-block file reads above threshold** → forces sub-pattern A
  routing automatically. The hook can decide cleanly from the tool
  input alone — it knows the path and the size.
- **Discipline (not hook) for fan-in synthesis and judgment-heavy
  tasks** → sub-pattern B/C calls are semantic state, not file paths.
  No PreToolUse hook can decide them from a `Read` / `Bash` payload.

The shipped product inherits this division: `omcp install --hooks`
ships the file-read gate; it does **not** try to strong-arm sub-
pattern B/C tasks into the bridge.

### 1.2 oMLX cache vs frontier token saving — orthogonal axes

A failure mode that risks confusing both v0.7+ product copy and PA
reviewers: oMLX's hot cache and SSD cache *sound* like they save
tokens. They do not.

Empirical layer-by-layer check from the 2026-05-15 dogfood session:

| Layer | Stores | Saves |
|---|---|---|
| oMLX hot cache (RAM, 6 GB) | model weights + active KV | model cold-load latency |
| oMLX SSD cache (disk, 8.8 GB → 11 GB after today's session) | content-addressed KV / weights, 37 MB chunks under `~/.omlx/cache/{0-f}/` | prefill compute on prefix match |
| Bridge tool itself | nothing (it's just an MCP entry point) | **frontier tokens** — and only when sub-pattern A applies (`source_uri`, raw bytes never enter frontier context) |

Concrete numbers from the 2026-05-15 session: 2 bridge `extract`
calls, footer-reported frontier saving ~1100 tok total, came
entirely from `source_uri`. Cache hit/miss did not change that
number. The "10-15 % frontier saving share" observed in the session
is bounded by **sub-pattern A trigger frequency**, not cache hit
rate.

**Implication for v0.7+ docs and PA review**:

- Cache benefits go to *task throughput / wall-time*, on the local
  machine, where there is no billing.
- Hook enforcement (gating sub-pattern A entry) is the only mechanism
  that affects *frontier billing*.
- Market and document the two separately. Don't let "8.8 GB cache!"
  get conflated with the frontier-saving story.

### 1.3 Install UX is the primary v0.7+ surface

The PM catch on 2026-05-18 expanded the v0.7+ scope. The 2026-05-15
framing positioned hooks as the v0.7+ surface and treated the rest
of the install path (Homebrew taps, model downloads, MCP config JSON
edits) as "user does these manually". That framing is wrong: every
step the user does manually is a step they can do wrong, postpone,
or skip — and a step that costs adoption.

The correct framing: **`omcp install` is THE v0.7+ feature.** It
delivers a one-line user experience that wires up everything:

```
$ npx local-mcp-toolbelt install
✓ Homebrew detected
✓ Installing oMLX via brew … (or detected, version 0.X)
✓ Starting oMLX service
✓ Downloading models (B + C, ~7.5 GB) … or skip with --no-models
✓ Detected Claude Desktop — wrote config entry to
  claude_desktop_config.json
✓ Wrote bridge-enforcement hook to .claude/settings.json
Done. Try: ask Claude "summarize /tmp/giant.log".
```

Hook installation (the original 2026-05-15 scope) is a sub-feature
inside this — equivalent to `omcp install --hooks` or part of the
default install bundle.

## 2. Candidate scope (subject to PA + Auditor)

### 2.1 `omcp install` — main entrypoint

Single command, idempotent, reversible:

```
omcp install                    # full: brew + service + models + config + hooks
omcp install --no-models        # skip 7.5 GB download
omcp install --no-hooks         # skip hook wiring
omcp install --models           # only download missing models
omcp install --config           # only write MCP client config
omcp install --hooks            # only wire bridge-enforcement hook
omcp install --check            # dry-run: list what would be done
omcp install --verify           # post-install smoke (initialize + tools/list)
omcp uninstall                  # reverse: drop config + hook entries
                                #   (keep models — explicit `--purge-models`)
```

### 2.2 What each sub-step does

- **brew check + install**: detect `brew`; then `brew tap
  jundot/omlx && brew install omlx` if missing. Surface clear error
  if Homebrew unavailable (do NOT auto-install Homebrew — too
  invasive).
- **service start**: `brew services start jundot/omlx/omlx`. Wait
  for oMLX to answer `/v1/models` (with timeout).
- **models**: existing `download-models` logic. Default B + C
  (~7.5 GB). Prompt user with size estimate before downloading
  unless `--yes` or stdin not a TTY. `--tiers B,C,D` for power
  users who want 14B.
- **MCP client config**: detect installed clients by config-path
  presence:
  - Claude Desktop: `~/Library/Application Support/Claude/claude_desktop_config.json`
  - Cursor: `~/Library/Application Support/Cursor/User/globalStorage/...`
  - Cline: VS Code extension config
  - Zed: `~/.config/zed/settings.json`
  For each detected client, write/merge a `local-mcp-toolbelt`
  entry using `npx local-mcp-toolbelt serve` as the command (so the
  user doesn't need a global install).
- **hooks**: write `.claude/settings.json` `PreToolUse` entry
  pointing at the shipped `enforce-bridge.sh` (now in
  `node_modules/local-mcp-toolbelt/hooks/`). Sentinel-tagged so
  `omcp uninstall` can remove only what was installed.

### 2.3 Hook script as shipped artefact

- **`hooks/`** directory in the published npm package containing
  `enforce-bridge.sh` (generic, env-var-driven — see current
  prototype at `.claude/hooks/enforce-bridge.sh` in this repo).
- **`hooks/README.md`** documenting the env-var knobs:
  `OMCP_HOOK_THRESHOLD_BYTES`, `OMCP_HOOK_ANALYSIS_THRESHOLD_BYTES`,
  `OMCP_HOOK_ANALYSIS_PATHS`, `OMCP_HOOK_DATA_EXTENSIONS`,
  `OMCP_HOOK_EXTRA_ALLOWED_PREFIXES`.

### 2.4 Verification

- `omcp install --verify` runs the same checks as
  `scripts/cross-client-smoke.mjs` (initialize + tools/list + a
  representative tool round-trip).

## 3. Out-of-scope (for this proposal)

- **Auto-install Homebrew**: too invasive. If `brew` is missing,
  surface a clear error and link to https://brew.sh.
- **Non-Apple-Silicon support**: out of scope — oMLX/MLX is Apple
  Silicon only.
- **Model auto-update**: model files are immutable per release;
  users opt into updates explicitly (`omcp install --models
  --force-redownload`).
- **MCP clients without a config-file surface**: clients that
  don't read a config file have to be configured by the user some
  other way. Document, don't try to automate.
- **Host-side enforcement for clients with no hook mechanism**:
  some MCP clients have no PreToolUse equivalent. Possibly
  addressed by an in-server enforcement signal in a later release
  once SEP-2663 Tasks stabilises.
- **Per-session adaptive hook thresholds**: static env vars are
  sufficient.
- **Telemetry / metric collection**: not selling a service.

## 4. Open questions for PA + Auditor

1. **Model download consent UX**: 7.5 GB is non-trivial. Prompt
   y/n? Default to yes if `--yes` or non-TTY? Or default skip and
   require explicit `--models`?
2. **Config merge strategy when entry exists**: if user already has
   `local-mcp-toolbelt` in their MCP config (different version, or
   manual install), do we overwrite, leave, or prompt?
3. **Hook sentinel format**: how to mark our hook entry so
   `omcp uninstall` can identify it? JSON comment-equivalent? Magic
   string in `command`?
4. **Multi-client config**: if user has Claude Desktop + Cursor
   both installed, write to both by default or just primary?
5. **Cross-platform fail-fast**: install command should fail-fast
   cleanly on Linux / Windows with a clear message (oMLX is Apple-
   Silicon only).
6. **`npx` vs `npm install -g`**: docs prefer `npx
   local-mcp-toolbelt install` to avoid global install drift.
   Verify that the npx-spawned bin can write to user-level config
   paths without permission issues.
7. **POSIX-sh-only vs Node hook implementation**: POSIX-sh is one
   less dep but harder to test; Node ships with the package anyway.
8. **Upgrade path if a user has customised their hook**: we should
   not silently overwrite; the sentinel should let us detect
   "this hook is ours, original" vs "this hook has been modified".
9. **SEP-2663 will eventually add server-side capability
   negotiation** — does the enforcement model collapse into that,
   or stay client-side?
10. **Interactive vs flag-driven by default**: `omcp install` no
    args → interactive prompts (better for adoption), `--yes` for
    CI / scripts? Or always non-interactive?

## 6. Why not just "tell users to do this themselves"

Four reasons:

1. **Soft-rule pattern always fails**: the whole reason the in-repo
   enforcement hook was needed is that *the rule alone doesn't
   work*. The same logic applies to install steps — asking users to
   read docs and run `brew install` + edit JSON + copy a shell
   script is the same honour-system failure mode applied at the
   install layer.
2. **Each client has its own config schema**: users shouldn't have
   to learn it; the install command handles the impedance.
3. **Updates**: when the hook gets a CVE fix or a new band, or
   models get retired, users on honour-system installs stay
   broken. Shipped + versioned is the only way to fix that.
4. **Adoption funnel**: each manual step drops some percent of
   users. A 6-step install yields a fraction of users vs a 1-step
   install. This determines whether the project gets used at all.

## 7. Not yet decided

- Release vehicle (own v0.7 vs ride a v0.6.x).
- Whether `omcp install` prompts interactively by default or uses
  flags. (Default y/n probably best for adoption; `--yes` for CI.)
- Whether to ship the bridge-enforcement hook as default-on or
  default-off. (Default-on matches the v0.6.0 PM stance:
  enforcement IS the product.)
- Whether `omcp install --verify` runs Tier B inference or just
  stdio handshake. Real inference verifies the full stack but
  costs ~10-20 s + a model load.
