## 1. What this release delivers

Two coupled concerns shipped together because they unblock each other:

1. **Async polling triad** — three new MCP tools (`enqueue_job`,
   `check_progress`, `read_job_result`) that let any MCP client dispatch
   long-running work without hitting Claude Code's hardcoded 60 s MCP
   wall (Fact 1 in synthesis). Universal pattern; works on Claude
   Desktop, Cursor, Cline, OpenClaw, Zed equally.

2. **Per-tool `thinking` flag** — all existing tools (`summarize`,
   `classify`, `extract`, `transform`, `summarize-long`,
   `summarize-long-chunked`, `diff-semantic-index`) gain an optional
   `thinking: 'on'|'off'|'auto'` parameter. `'auto'` (the default)
   resolves to a server-side per-tool registry based on Phase 1 +
   thinking-eval data. Long-form thinking is what makes async useful;
   short-form thinking already fits sync.

Both concerns trace to the same insight from the R2 brainstorm: the
60 s wall is the **necessary precondition** to safely default
thinking-on, which is the **direct quality lever** Phase 1 eval data
points at. Quality is the real bottleneck (mean 2.5/5 no-think), not
latency or RAM.

## 2. Out of scope for v0.6.0

- **Filesystem-only return channel without polling** — rejected per gem
  R2: sandboxed clients (Cursor in some configs) may lack `file://`
  read access. `read_job_result` returns inline content as the primary
  path; file paths are an optional fallback for very large results.
- **`auto` as a runtime LLM-decides flag** — `'auto'` is a deterministic
  alias for the per-tool registry. Future "model decides if thinking
  helps" is a v0.7+ research question, not v0.6.0.
- **Bash-wait pattern as a required path** — opt-in optimization for
  Claude Code only via `wait_command` field on the `enqueue_job` return.
  Cross-client correctness does NOT depend on Bash being available.
  3/4 R2 voices vote cut Bash-wait if scope pressures; we keep it as
  optimization but commit to ship working without it.
- **Migration off `wait_for_job` (v0.3.0)** — the old long-poll tool is
  deprecated but stays for one minor version. `enqueue_job` /
  `check_progress` / `read_job_result` are the canonical path in v0.6.0;
  v0.7.0 removes `wait_for_job`.
- **Server-initiated push / SSE** — explicitly out of scope. MCP
  transport is request-response; we don't extend it.
- **Tier D demotion based on no-think eval** — Phase 1 showed 14B mean
  2.50 ≈ 4B mean 2.54. Thinking-mode subset eval showed transform +1.0
  on 14B. v0.6.0 keeps Tier D as opt-in with thinking-on default; the
  decision to demote is deferred until thinking-on full eval lands.

## 3. New MCP tools

### 3.1 `enqueue_job`

```
enqueue_job(
  tool: string,                    // any existing tool name (not enqueue_job itself)
  args: Record<string, unknown>,   // forwarded to the underlying tool
  opts?: {
    thinking?: 'on' | 'off' | 'auto'   // default 'auto' (see §4)
  },
)
→
{
  job_id: string,                  // nanoid(10), path-safe
  enqueued_at: string,             // ISO8601
  result_uri: string,              // file://~/.local-mcp/jobs/<id>.json
  est_ms?: number,                 // server estimate based on input size + tool
  wait_command?: string,           // present iff client has 'bash' capability
}
```

Side effects:
- Persists `{status:'queued', tool, args, thinking_resolved, enqueued_at}`
  to `~/.local-mcp/jobs/<id>.meta.json`.
- Worker picks up job; runs underlying tool with the resolved thinking
  setting; writes result via atomic rename to `<id>.json`.
- Dedups identical concurrent enqueues by SHA-256 of canonical-sorted
  `(tool, args, thinking_resolved)`. v0.3.0 dedup logic re-used.

### 3.2 `check_progress`

```
check_progress(job_id: string)
→
{
  status: 'queued' | 'running' | 'done' | 'error',
  progress?: number,               // 0–100, only when meaningful (chunked tools)
  eta_s?: number,                  // server's remaining-time estimate
  error?: string,                  // present iff status === 'error'
}
```

Instant return (<100 ms wall). Polled by clients without Bash. Each call
is one MCP roundtrip; safe to call repeatedly. Suggested polling
cadence: every 2 s for the first 60 s, every 5 s thereafter.

### 3.3 `read_job_result`

```
read_job_result(job_id: string)
→
{ content: string }                // inline result text, ≤ 1 MB default
| { file_path: string,             // for results > 1 MB; client reads via file:// or its own means
    bytes: number }
| { error: string }
```

Inline by default. Threshold configurable via `OMCP_INLINE_RESULT_MAX_BYTES`
(default 1 048 576). Above the threshold, the bridge persists to disk and
returns the path — but most bridge tasks (summary, classify, extract)
produce ≤ 50 KB output, so file path is rarely used.

## 4. Modified existing tools — `thinking` flag

All seven existing tools (`summarize`, `summarize-long`,
`summarize-long-chunked`, `classify`, `extract`, `transform`,
`diff-semantic-index`) gain:

```
  thinking?: 'on' | 'off' | 'auto'   // default 'auto'
```

Resolution order:
1. Per-call value passed to the tool overrides everything.
2. Per-tool env var `OMCP_THINKING_<TOOL_NAME>` (e.g.
   `OMCP_THINKING_SUMMARIZE=off`) — uppercased + underscored.
3. Global env var `OMCP_THINKING_MODE=on|off` — affects all tools.
4. Tool-registry default in `server.ts`:

| Tool | Default | Justification (Phase 1 + thinking-eval) |
|---|---|---|
| `classify` | on | structured output; reasoning helps decide |
| `extract` | on | schema-bound; reasoning helps comply |
| `transform` | on | thinking-eval transform +1.0 score lift |
| `diff-semantic-index` | on | structured + reasoning task |
| `summarize` | off | prose; thinking-eval summarize −0.5 score loss |
| `summarize-long` | off | same |
| `summarize-long-chunked` | off | same |

Implementation: `MlxHttpBackend.chat()` already supports
`OMCP_THINKING_MODE=on` (env var added in commit pending in v0.5.1's
session). v0.6.0 extends to per-tool granularity by computing the
effective value in `src/mcp/server.ts` per tool call and passing it
into ChatOptions as a new field `disableThinking: boolean`. The
backend honors that field and appends/omits the `/no_think` suffix
accordingly.

## 5. Cross-client policy

| Client | Behavior |
|---|---|
| Claude Code | Use `enqueue_job` for any task expected to exceed 60 s. Use `wait_command` (returned by `enqueue_job`) via the built-in Bash tool to collapse poll+read into one Bash invocation. |
| Claude Desktop | 240 s request timeout. May use sync tools for most cases. For >240 s tasks, `enqueue_job` + poll `check_progress` works without Bash. |
| Cursor / Cline / OpenClaw / Zed | Configurable timeouts; same async triad as Claude Desktop. No Bash dependency. |

Capability detection: `wait_command` is returned ONLY when the client
advertises `bash` capability (MCP capability handshake) OR the env var
`OMCP_ASSUME_BASH_CLIENT=1` is set. Defaults: no advertisement → no
`wait_command` field in response.

## 7. Success criteria (measurable)

1. **Quality lift demonstrated**: re-run of Phase 1 eval (5 fixtures × 3
   trials × 3 tiers = 45 trials) with v0.6.0 defaults produces:
   - Mean score ≥ 4.0/5 on at least 2 of the 3 tiers (B or C or D)
   - Transform task specifically: ≥ 4.0/5 on Tier D
   - Summarize tasks: no regression vs Phase 1 baseline (no-think) — the
     per-tool default off must protect prose quality.
2. **Cross-client smoke**: `enqueue_job` → `check_progress` →
   `read_job_result` triad works against Claude Desktop (config swap) for
   at least one fixture per tool kind.
3. **No user-visible failures from oMLX abort**: 100 sequential
   thinking-mode trials across the eval matrix; circuit-breaker telemetry
   (commit 994f01c) shows ≤ 5 % retry rate; all 100 trials return a final
   result (retry-masked or first-try-ok).
4. **60 s wall not binding**: ≥ 90 % of thinking-on calls in the eval
   matrix complete in a single `enqueue_job → check_progress (once) →
   read_job_result` sequence — i.e. the work fits in one poll cycle.
   The async-job pattern handles the remaining ≤ 10 %.
5. **151 + 12 = 163 unit tests pass**.

## 8. Risks + mitigations

| Risk | Mitigation |
|---|---|
| `'auto'` interpretation drifts toward "model decides" | Locked in §2 as alias for registry; no runtime LLM decision in v0.6.0 |
| Sandboxed clients lack file:// access | `read_job_result` returns inline content as primary; large results spill to file with explicit threshold |
| Anthropic narrows Bash tool 10 min cap | `wait_command` is opt-in via capability; universal polling still works |
| oMLX SIGABRT under thinking-on (heavier decode = more Metal command buffers) | Circuit-breaker already in place; stderr telemetry monitors rate; success-criterion #3 caps retry-rate at 5 % |
| Per-tool registry defaults wrong for some inputs | Per-call override + env var override; document the resolution order |
| `wait_for_job` consumers break | Deprecation banner stays for one minor version; v0.6.0 README documents the migration |

## 10. Sequence with v0.5.x

v0.5.1 (current main) ships independently of v0.6.0. v0.5.1's
contributions (HEAD upgrade + circuit-breaker + telemetry + Phase 1
baseline + `OMCP_THINKING_MODE` env var) all stay; v0.6.0 builds on
them. No revert of v0.5.x work is part of v0.6.0.

If v0.6.0 is abandoned mid-development, v0.5.1 still ships a
better-than-v0.5.0 product (the circuit-breaker alone is meaningful
stability gain).
