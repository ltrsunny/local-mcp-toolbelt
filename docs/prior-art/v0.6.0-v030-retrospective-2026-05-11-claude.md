# Prior Art Candidate — v0.3.0 in-house retrospective

**Date:** 2026-05-11
**Author:** Claude (in-house retrospective; not external prior art)
**Status:** PA Review candidate for v0.6.0 60s-wall solution

## What v0.3.0 actually shipped

Three MCP tools + supporting infrastructure (`packages/core/src/jobs/`):

| Tool | Behavior | Code |
|---|---|---|
| `enqueue-job(tool_name, args)` | Returns `{job_id}` <1s. Dedups by `hash(tool_name + canonical(args))`. Persists `{status:'queued',…}` to `.memory/jobs/<id>.json`. | `registry.ts:enqueue` |
| `wait_for_job(job_id, max_wait_ms=50000)` | Event-driven block ≤50s via JobRegistry EventEmitter. Returns `still_running` if timeout, `done` with result body if finished. Proper unsubscribe on each abort cycle. | `registry.ts:onUpdate` |
| `read_job_result(job_id)` | Reads persisted Memory Bank entry (`.memory/jobs/<id>.md` + JSON sidecar). | `store.ts:readResult` |

**Lifecycle state machine**: `queued → running → done | failed`. Orphan recovery on bridge startup marks any leftover `queued/running` jobs as `failed` with reason `'bridge restarted'`.

**Persistence**: file-based, `.memory/jobs/`, TTL 7 days, GC on start.

**Dedup**: 16-char SHA-256 prefix of canonical-sorted args; identical concurrent calls return the same `job_id`.

## What worked (kept as-is for v0.6.0)

- **File-based persistence**. Survives bridge restarts cleanly. No DB dependency.
- **Hash-based dedup**. Eliminates frontier busy-loop accidents (Claude re-calling identical args ten times in a confused retry loop).
- **Event-driven `wait`**, not poll-loop. Avoids the "burn frontier turns waiting" failure mode that drove the original decision.
- **Orphan reconciliation**. Crash-safe restart semantics.
- **Atomic write pattern**. Implicit in `writeMetadata` → temp + rename.

## What didn't work (where v0.6.0 diverges)

1. **`wait_for_job` is itself bound by the 60s MCP wall.** Cap is 50s; client still re-calls. The very pattern the architecture was supposed to obsolete (Claude polling) is just reintroduced one layer down.
2. **No fast-approx path.** For 80% of tasks where a Tier B answer in 30s is shippable, Claude must still wait the full job. This is the strongest finding from the 2026-05-11 brainstorm.
3. **No streaming progress.** `wait_for_job` returns `still_running` (boolean) — no chunk-level visibility. Long jobs are opaque until done.
4. **Resurfaces 60s wall on `read_job_result` for huge outputs.** Reading a 5 MB summary back through MCP risks framing/encoding cost approaching the wall.
5. **Real-world usage <5%** during v0.2.0/v0.3.0 cycles. The pattern is correct but the UX cost (manage retry loop, no fast feedback) makes Claude default to inline tools.

## Lessons for v0.6.0

1. **Keep**: file-based persistence, hash-dedup, atomic writes, orphan recovery, EventEmitter machinery.
2. **Replace**: `wait_for_job` → `check_progress` (lightweight status only, no waiting; encourages Claude to use Bash for actual sleep)
3. **Add**: `fast_approx` opt-in path on `enqueue-job` (returns approximation + refined `job_id` in one call)
4. **Add**: `wait_command(job_id)` returning a ready-made Bash one-liner Claude can paste into its Bash tool (10-min cap >> 60s MCP wall).
5. **Streaming progress**: emit partial-chunk count in metadata `progress` field so `check_progress` returns useful detail, not just bool.

## Cost framing

v0.3.0 was 5-7 dev-days (per the brainstorm estimate; actually delivered in ~3 commits). The infrastructure is mostly intact and re-usable — v0.6.0 is largely an **API ergonomics refactor on top of working internals**, not a from-scratch rebuild.

Estimated v0.6.0 implementation cost: **S-M**, mostly:
- Delete `wait_for_job` (or alias for back-compat)
- Add `check_progress`, `wait_command`, `fast_approx` parameter
- Per-tool `prepare_fast_approx()` hook (optional)
- Migration note + telemetry diff
