# Prior Art: MCP + Long-Running / Async Patterns (collected 2026-05-11)

Below are 5 real open-source repositories (MCP or analogous inference servers) that address long-running requests and asynchronous result delivery. Each entry lists repo URL, the approach used for long runs, why the approach was chosen (from docs/PRs/issues), and tradeoffs reported by maintainers or in issue history.

---

## 1) modelcontextprotocol/experimental-ext-tasks
Repo: https://github.com/modelcontextprotocol/experimental-ext-tasks

Approach: protocol-level "Tasks" extension — server-created durable Task objects with unique task IDs, server-side state machine, and explicit task API methods (tasks/get, tasks/update, tasks/cancel). Pattern is call-now / fetch-later: a tools/call can return a polymorphic resultType: "task" instead of an immediate final result; the client polls or requests the task resource to get eventual result.

Why: the README and SEP-2663 PR explicitly define Tasks to represent expensive computations, batch work, and to integrate external job/workflow APIs. The extension was introduced to provide a standard, interoperable primitive for deferred retrieval that fits MCP's JSON-RPC semantics.

Tradeoffs / issues: authors intentionally keep this experimental — tradeoffs shown in the PR discussion: complexity of durable state (who stores task state), lifecycle semantics (garbage collection/TTL), and client/server capability negotiation (server decides per-request whether to materialize a task). The approach shifts complexity into server-side durable storage and task lifecycle policies; documentation and governance are still evolving.

## 2) kEND/mcp_sse (MCP over Server-Sent Events)
Repo: https://github.com/kEND/mcp_sse

Approach: transport-level solution using SSE for long-lived connections. The library implements MCP over SSE with an EventSource-style endpoint for server->client events and a separate /message endpoint for client->server JSON-RPC. It includes keepalive/ping, session IDs, and connection management.

Why: SSE maps well to MCP’s JSON-RPC + event-centric flow: long-running tools can emit intermediate events and final results over the open SSE channel. The README documents keepalive pings and session management as first-class concerns; authors chose SSE for simplicity (browser-friendly, easy to consume) and for stable unidirectional server push.

Tradeoffs / issues: SSE requires connection liveness (intermediaries or proxies can drop idle connections), so configurable keepalives and reconnection logic are necessary. SSE is unidirectional (server->client) and doesn't replace the need for a request channel; the repo documents the need for a paired /message endpoint and session negotiation. Operationally, SSE-based servers must manage many long-lived connections and scale ramifications (backpressure, proxy timeouts), and the README explicitly exposes keepalive configuration as a primary operational knob.

## 3) modelcontextprotocol/servers (reference servers + frameworks)
Repo: https://github.com/modelcontextprotocol/servers

Approach: curated collection of reference MCP server implementations and transport/framework examples. The repo points to multiple patterns used by MCP servers: (a) streaming transports (SSE/websockets), (b) example reference servers illustrating call-now / fetch-later and resource referencing, and (c) extension experiments (e.g., tasks). It also surfaces frameworks (FastMCP, mcp_sse, Anubis) used to implement long-running tools.

Why: the servers repo is the canonical reference for how MCP features are typically implemented. It documents example server modules (Filesystem, Fetch, Everything) and links to transport implementations; the reference nature motivated conservative, interoperable patterns rather than inventing ad-hoc approaches.

Tradeoffs / issues: by design these are reference/educational servers rather than production-grade frameworks — maintainers warn about operational considerations. Issues across the referenced projects reveal recurring tradeoffs: either rely on streaming transports (which require connection keepalive and scale planning) or push complexity into server-side durable state (task stores, TTLs) and polling. The repo collates guidance but leaves production choices to implementers.

## 4) vllm-project/vllm
Repo: https://github.com/vllm-project/vllm

Approach: inference-serving library with OpenAI-compatible API, streaming outputs, and disaggregated prefill/decode/batching. vLLM focuses on continuous batching, streaming responses to clients, and disaggregated worker topologies (engine workers, schedulers) to keep latency low while allowing long-running generations. It exposes streaming endpoints (chunked responses/SSE-like behavior or websockets/gRPC in some deployments) and implements mechanisms to prefill, pipeline, and speculative-decode for heavy workloads.

Why: vLLM’s README and docs emphasize throughput and streaming as the primary mechanism to avoid client timeouts — by delivering tokens or intermediate outputs incrementally, clients do not need a monolithic long request. The project also invests in disaggregation and fault isolation so individual long operations do not stall the serving fabric.

Tradeoffs / issues: the vLLM issue tracker shows numerous hang/timeout and worker-deadlock reports when disaggregation/parallelism gets complex (Ray/worker channel timeouts, engine hang cases). Maintainers face tradeoffs between offering streaming low-latency partial results (good for client timeouts) and the increased complexity of scheduling, backpressure, and correctness across disaggregated workers. Many open issues reference race conditions, Ray/RPC timeouts, and the need for robust retry/health endpoints.

## 5) ggml-org/llama.cpp (llama-server)
Repo: https://github.com/ggml-org/llama.cpp

Approach: C/C++ inference runtime with an optional OpenAI-compatible `llama-server` mode. Server examples provide HTTP endpoints and streaming-style outputs (HTTP chunking / WebSocket examples in examples/), enabling incremental output delivery rather than a single long-blocking response. The project includes a `llama-server` and WebUI that are explicitly used to serve model generations.

Why: project maintainers chose streaming/chunked HTTP and lightweight RPC examples because they are portable, minimal, and avoid forcing the client to wait for the entire generation. The README and server docs point to launching a `llama-server` that supports online generation and router modes for model loading.

Tradeoffs / issues: issue history shows server hangs, deadlocks, and cases where streaming endpoints block; maintainers continuously fix edge cases (worker deadlocks, router aborts). Tradeoffs are operational: C/C++ server stability, edge-case concurrency bugs, and the difficulty of implementing safe preemption / cancellation in low-level runtimes. The pattern reduces client-blocking but shifts complexity into the server’s concurrency and I/O handling.
