# Long-Running RPC Patterns Under Fixed Timeouts

## 1. Callback URLs (Webhooks)
**Description**: Server calls back to client-provided endpoint upon completion
- **Systems**: Temporal, AWS Step Functions, Cadence, Stripe webhooks, PayPal IPN
- **Tradeoff**: Reduces latency for client; increases complexity and failure modes; requires state management
- **Avoid when**: Client cannot expose HTTP endpoints or handle inbound connections

## 2. Polling with HTTP 202 + Location
**Description**: Server returns 202 Accepted with status endpoint; client polls for result
- **Systems**: REST APIs, AWS EC2 instance launches, S3 Multipart uploads, Google Cloud operations
- **Tradeoff**: Simple client implementation; high latency and server polling load; requires client state tracking
- **Avoid when**: Latency is critical or request volume is high

## 3. Async Request with Result Storage
**Description**: Server stores result asynchronously; client polls result endpoint
- **Systems**: AWS Lambda async invokes, Google Cloud Functions, Azure Functions
- **Tradeoff**: Eventual consistency; complexity in result storage and cleanup; client must handle polling
- **Avoid when**: Immediate client feedback is required

## 4. Server-Sent Events (SSE)
**Description**: Server pushes updates over persistent HTTP connection
- **Systems**: Real-time dashboards, stock tickers, notification systems
- **Tradeoff**: Low latency updates; connection management complexity; limited browser support
- **Avoid when**: Client lacks persistent connection capability

## 5. Client-Controlled Long Polling
**Description**: Client holds connection until server has result or timeout
- **Systems**: Chat applications, real-time notifications, gRPC server-streaming
- **Tradeoff**: Resource consumption during waits; connection scalability; server complexity
- **Avoid when**: Persistent connections are unavailable or unreliable

## 6. Periodic Status Polling
**Description**: Client repeatedly polls server for operation status
- **Systems**: Batch processing systems, Celery, BullMQ, background job processors
- **Tradeoff**: Latency vs. resource usage tradeoff; client complexity for polling logic
- **Avoid when**: Low-latency feedback is required

## 7. Result Pagination/Cursor Pattern
**Description**: Paginated result retrieval with continuation tokens
- **Systems**: Twitter API, GitHub API, database query results
- **Tradeoff**: Efficient resource usage; client complexity for state management
- **Avoid when**: Operations must be atomic or non-continuable

## Recommendation

For 60s timeout + 30-300s work: **Client-Controlled Long Polling** and **Async Request with Result Storage** patterns apply most directly. **Callback URLs** work if client can receive callbacks. Combine **Polling** with **Result Storage** for robust LLM integration scenarios.
