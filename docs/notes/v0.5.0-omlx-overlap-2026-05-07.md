# oMLX vs local-mcp-toolbelt — overlap analysis (2026-05-07)

The user asked whether oMLX (`jundot/omlx`) subsumes this project. Short
answer: no — they live at different layers, complementary not competitive.

## Layer separation

| Concern | oMLX | local-mcp-toolbelt |
|---|---|---|
| Inference engine | ✅ MLX backend; KV-cache; multi-model | ❌ delegates |
| HTTP API | ✅ `/v1/messages` + `/v1/chat/completions` | ❌ |
| MCP tools registered with Claude Code | ❌ | ✅ 6 tools |
| Source-URI privacy (`file://`, `http://`) | ❌ | ✅ |
| Prompt-injection defender | ❌ | ✅ Layer-1 + Layer-2 |
| Map-reduce chunking | ❌ | ✅ |
| Grammar-constrained output | ❌ (`json_object` only) | ✅ via llama.cpp GBNF |
| Async-job pattern (60 s wall escape) | ❌ | ✅ |
| Tier routing (B/C/D) | ❌ | ✅ |

**Conclusion**: oMLX is a Tier-D inference *backend* for us; it does not
replace MCP-tool semantics or the privacy/defender/grammar layers.

## Two oMLX features that *look* like overlap

### 1. `claude_code` settings section

oMLX `~/.omlx/settings.json` has a `claude_code` block with
`opus_model`/`sonnet_model`/`haiku_model` slots. The use case: route
Claude Code's *base inference* (messages API calls) to a local model.

This is unrelated to MCP tools. Claude Code can simultaneously:
- talk to Anthropic API for its primary chat loop (or oMLX, if user
  routes there);
- talk to MCP servers (us) for delegated tools.

The two paths don't compete — they just happen to share a project name.

### 2. oMLX `/v1/mcp/*` endpoints

oMLX exposes `/v1/mcp/execute`, `/v1/mcp/servers`, `/v1/mcp/tools` for
*non-MCP-aware HTTP clients* to call MCP tools through oMLX as a gateway.
That gateway USES our MCP server, doesn't replace it.

Specifically: an OpenAI-API client could POST to oMLX and have oMLX
internally call local-mcp-toolbelt's tools and return the results.

## What we should NOT build

Given oMLX exists, we should **not** build:

- A local-mcp-toolbelt analog of oMLX's KV-cache persistence (oMLX does
  this for us when on Tier D)
- A direct-message API in local-mcp-toolbelt (out of scope; oMLX already
  has `/v1/messages`)
- A "Claude Code base inference" replacement (that's oMLX's lane)

## What we should build (Tier-D era roadmap)

1. **Document Tier-D opt-in** properly in README + `packages/core/README.md`
   (currently only in CLAUDE.md and scope memo).
2. **Investigate grammar enforcement on oMLX**. If oMLX adds `json_schema`
   Structured Outputs, we can promote `classify`/`extract` to Tier D with
   real schema constraints.
3. **Quantify KV-cache benefit** — oMLX persists prefix cache across
   requests. For repeated-prefix tools (e.g. summarize-long-chunked
   reusing the same large source), this could be a major latency win.
4. **Stop building inference plumbing**. oMLX is a stable, Apache-2.0,
   community-supported piece of infra. Use it; don't reimplement.

## Decision

local-mcp-toolbelt continues. oMLX is consumed as the Tier-D backend.
No project-wide architectural change required.
