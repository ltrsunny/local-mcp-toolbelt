# v0.6.0 brainstorm — solving the 60s MCP wall

**Date:** 2026-05-11
**Mode:** R1 → R2 mutual challenge (4 voices: Claude, Gemini, Copilot, nv_pro/Qwen3.5-397b)
**Outcome:** Strong convergence on filesystem-as-return-channel + heartbeat add-on; 2-2 split on fast-approx; this doc seeds the v0.6.0 scope memo

## Problem framing

Claude Code MCP request timeout is hardcoded ~60s wall-clock (anthropics/claude-code#5221, #22542). v0.3.0 added async-job pattern (enqueue-job → wait_for_job → read_job_result) but `wait_for_job` itself has the same 60s wall, capping each poll at 45s and forcing Claude to manage the retry loop.

## R1 — independent proposals (3 per voice, 12 total)

### Claude (claude-opus-4-7)
1. **C1 filesystem return channel** — bridge writes result file; Claude uses own Read tool. wait_for_job IS the bug.
2. **C2 per-tool chunk variants** — expose `chunk-summarize(idx)`, `chunk-reduce()`; Claude orchestrates
3. **C3 result hash + idempotent re-submit** — every call returns done/pending/queued; dedup

### Gemini (gemini-3.1-pro-preview)
1. **Macro-Delegation** — autonomous local sub-agent owns the loop; returns file path immediately
2. **Pipelined Continuation** — 45s synchronous + continuation_token; Claude actively loops
3. **Shell-Mediated Blocking** — return a bash `while [ ! -f ]` one-liner for Claude's Bash tool

### Copilot
1. **Two-tier pipeline** — fast-approx (Tier B, always <60s) immediately; refined in background
2. **Resumable map-reduce orchestration** — split into <60s chunks; persist artifacts
3. **Backend snapshot/resume** — checkpoint oMLX generation state across calls

### nv_pro (Qwen3.5-397b-a17b)
1. **Chunked streaming via partial results** — partial output across 45s windows
2. **Pre-chunking input** — sub-slice inputs to <60s units before processing
3. **Internal timeout handoff** — server tracks progress, early-return near 60s

## R2 — mutual challenge (each voice rates others)

| Voice | Strongest cross-voice | Most overrated | Final architecture |
|---|---|---|---|
| Claude | Copilot two-tier | nv_pro #1 | Two-tier + filesystem + bash handoff |
| Gem | Claude C1 filesystem | Copilot two-tier | C1 + Gem #2 pipelined + heartbeat tool |
| Copilot | Copilot two-tier (own) | Gem #3 shell-blocking | Two-tier + map-reduce checkpoint + optional filesystem |
| nv_pro | Claude C1 filesystem | Copilot two-tier | C1 + Gem #1 sub-agent |

## Convergence analysis

**Filesystem-as-return-channel: 4/4 in final architecture.** All voices, even ones that voted differently on which idea is "strongest cross-voice," put filesystem return in their final ship architecture.

**Two-tier (fast-approx): 2 for, 2 against.** Champions (Claude, Copilot) say it eliminates the wait entirely; opponents (Gem, nv_pro) call out:
- Gem: "prefill of 100k-token doc breaches 60s even on Tier B — fast-approx is fiction for the actual hard cases"
- nv_pro: "users act on the placeholder, then refined contradicts — bad UX"

**Heartbeat/progress signal: 2 voices add it.** Gem & Copilot both worry about client behavior (Claude reading incomplete file, hallucinating). A lightweight `check_progress(job_id)` MCP call gives Claude tangible progress without burning a real polling call.

**Sub-agent framing: 2 voices.** Gem #1 and nv_pro #4 both emphasize "bridge becomes autonomous peer, not dumb function." This is more philosophical than mechanical.

## Sharpest critiques (collected from R2)

**Against Claude C1 (filesystem return):**
- "Doesn't eliminate polling, relocates it" (Claude's own self-critique)
- "Cross-platform atomic rename semantics vary; expands threat surface (sensitive outputs on disk)" (Copilot)
- "What prevents Claude from reading incomplete file or looping on ENOENT forever?" (nv_pro)

**Against Copilot two-tier:**
- "Prefill of long doc breaches 60s even on Tier B" (Gem)
- "Users act on placeholder, refined contradicts → UX hazard" (nv_pro)

**Against nv_pro chunked-streaming:**
- "Vague restatement of existing wait_for_job pattern" (Claude)

**Against Gem shell-blocking:**
- "Depends on undocumented client terminal behavior" (Copilot)

## Synthesized v0.6.0 architecture

Combining the consensus + complementary add-ons:

### API surface

```
enqueue-job(tool, args, opts?: { fast_approx?: bool })
  → {
      job_id,
      result_path: "~/.local-mcp/jobs/<id>.json",
      fast_approx?: string,           // present iff opts.fast_approx
      approx_quality?: "rough"|"good",
      hint: "call check_progress(job_id), then read_job_result(job_id)"
    }

check_progress(job_id)
  → { status: "pending"|"done"|"error",
      progress: 0..100,
      eta_ms?: number,
      partial_chunks_done?: number }

read_job_result(job_id)       // existing; instant; reads result_path

wait_command(job_id)?          // OPTIONAL — returns bash one-liner
                               // for use in Claude's Bash tool (10min cap)
```

### Behavioral contract

- Bridge spawns worker; worker writes result via atomic rename
- Claude default flow: `enqueue` → `check_progress` (1-3 times) → `read_job_result`
- For tasks with meaningful fast-approx: Claude can act on `fast_approx` immediately, only fetch refined if needed
- For >5min jobs: Claude uses `wait_command` in Bash (10min Bash timeout >> 60s MCP timeout)

### What about indivisible bottlenecks?

Gem's killshot ("14B model load alone can blow 60s") is real but applies to a narrow case: cold-start of a tier not currently loaded. Mitigation: warm Tier B at MCP server startup; document that Tier D first-call is slow.

### What's deliberately dropped

- **wait_for_job** — replaced by check_progress (lighter) + read_job_result
- **Chunk-variant tools** (Claude C2) — too much state mgmt cost; map-reduce stays inside `summarize-long-chunked`
- **Backend state snapshot** (Copilot #3, nv_pro #3) — oMLX doesn't expose it

## Risk summary (from R2 attacks, retained)

| Risk | Source | Mitigation |
|---|---|---|
| fast-approx quality | Copilot R1 + nv_pro R2 | `approx_quality` label; eval covers approx-vs-refined gap |
| cross-platform atomic rename | Copilot R2 on C1 | document POSIX same-dev rename guarantee; reject jobs across mount points |
| Claude hallucinates from empty file | Gem R1 + R2 | `check_progress` is the strongly-hinted poll path, not direct Read |
| Bash-handoff timeout | Gem #3 | Bash 10min cap is documented (`timeout` param); fall back to MCP-poll if >10min |
| Sensitive output on disk | Copilot R2 | Job dir 0700; TTL 7d; signed payloads optional |

## Engineering cost

**M** (medium). Reuse existing async-job worker, atomic writes, and TTL logic from v0.3.0. New surface: `check_progress` tool, `fast_approx` opt-in path per tool, `wait_command` helper. Refactor `wait_for_job` → `check_progress` (or keep both during deprecation window).

## Post-brainstorm: v0.5.x hot_cache patch vs v0.6.0 structural fix

After the brainstorm we tested whether a simple oMLX config change could
move the goalposts enough to delay v0.6.0. Findings (2026-05-11):

### Hardware probe

- Mac: MacBookPro18,1, **16 GB unified memory**
- oMLX before any 14B call: 7.59 GB RSS (3 models registered in catalog)
- oMLX immediately after Qwen3-14B-4bit call: **1.90 GB RSS** ← dropped
- Insight: **MLX RSS is misleading.** Unified-memory model weights are GPU
  memory-mapped, not counted in process RSS. Wall-time is the real signal.

### `hot_cache_max_size` 4GB → 10GB experiment

Restart oMLX, then 3 consecutive identical 14B classify calls (50 tok cap):

| Run | State | Wall | Tokens out |
|---|---|---|---|
| 1 | Cold (first after restart) | **7338 ms** | 6 |
| 2 | Warm-1 | **1088 ms** | 6 |
| 3 | Warm-2 | **1061 ms** | 6 |

Hot cache cuts the cold-load tax of **~6.2 s** entirely. Warm 14B decode
is ~30 tok/s — much faster than the cold-data 12-16 tok/s in pre-2026-05-11
docs.

### Projected Tier D budgets (warm, short input)

| Task | Output cap | Projected wall | 60 s margin |
|---|---|---|---|
| classify | 200 tok | ~8 s | 52 s ✅ |
| summarize | 600 tok | ~22 s | 38 s ✅ |
| transform | 1200 tok | ~45 s | 15 s ⚠️ |
| **Long input (5K+ tok prefill)** | any | **prefill ~17 s + decode → 60 s+** | ❌ |

### Conclusion: patch is real, but does not retire v0.6.0

The hot_cache change expands Tier D's usable surface from "only classify"
to "classify + short summarize + short transform." That's a genuine win
for ~70 % of plausible Tier D tasks and ships TODAY with zero code.

But: **long-input + long-output cases still hit the wall.** Specifically
`summarize-long` and `extract` on documents > ~5 K tokens compound prefill
and decode past 60 s on 14B. No oMLX config makes that go away.

So v0.5.x and v0.6.0 are complementary, not alternative:

| Path | Solves | Cost |
|---|---|---|
| v0.5.x: `hot_cache=10GB` | Cold-start tax; expands Tier D's usable surface | 0 (config only) |
| v0.6.0: filesystem return + check_progress | The 60s wall for genuinely long jobs | M |

**Decision:** ship the hot_cache change in a v0.5.1 patch release (config
note + CHANGELOG line), keep v0.6.0 scope memo on the structural fix.
