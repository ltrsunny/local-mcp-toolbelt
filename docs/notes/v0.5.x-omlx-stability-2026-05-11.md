# v0.5.x oMLX stability finding — uncaught Metal exception causes abort

**Date:** 2026-05-11
**Severity:** P1 — kills inference server mid-request, but launchd auto-recovers
**Scope:** Affects all local-mcp-toolbelt v0.5.x users on oMLX 0.3.8.
**Authoritative artifacts:**
- `.claude/diagnostics/oMLX-crash-100611.ips`, `oMLX-crash-185307.ips`
- `.claude/diagnostics/upstream-bug-report.md` (draft ready for upstream)

## What we observed

Two `Python.app` crashes (oMLX runtime) on 2026-05-10 and 2026-05-11, both
SIGABRT, both with the same C++ throw path:

```
mlx::core::gpu::check_error(MTL::CommandBuffer*) → throws
  → __cxa_throw → std::__terminate → abort()
```

The crash report from 2026-05-11 shows MLX's GPU completion handler
detecting a Metal error and throwing. oMLX's Python wrapper has no
`catch` for this path, so any GPU-side error tears down the whole
`omlx serve` process. `launchd` restarts within ~1 s, but in-flight
HTTP requests are severed.

This is an **oMLX upstream bug**, not a local-mcp-toolbelt bug. Our
`MlxHttpBackend` cannot prevent it; we can only react to the dropped
connection.

## Triggers we identified (in order of impact)

1. **`hot_cache_max_size` ≥ 8GB on 16GB Mac** — drives memory pressure
   high enough that Metal allocations occasionally fail. We tested 10GB
   and triggered the crash within 17 s of bridge load. Rolled back to
   **6GB** as a safe default.
2. **Fast tier alternation** (4B → 8B → 14B in quick succession) — model
   swaps stress Metal heap. Phase 1 eval that walks all 3 tiers is the
   highest-risk workload.
3. **Concurrent HTTP requests under memory pressure** — the bridge's
   `summarize-long-chunked` map-reduce path fans out 2 chunks concurrently
   by default (`OMCP_CHUNK_CONCURRENCY=2`); under pressure, that
   doubles Metal-heap churn.

## Operational guardrails (apply NOW, before any upstream fix)

- [x] `hot_cache_max_size` ≤ 6GB on 16GB Macs (in `~/.omlx/settings.json`)
- [ ] Single-tier sessions: don't alternate models in the same minute if
      memory is tight; warm one tier at a time
- [ ] If eval is run, do it per-tier (single `--mlx-model`), not as a
      multi-tier matrix run
- [ ] Watch `vm_stat` Pages free and `sysctl vm.swapusage`; if free < 500
      MB OR swap > 3 GB, pause new bridge requests

## Decision: Stop the line on Phase 1 eval

Per the user's "D" option in the 2026-05-11 crash-decision branch: do not
run additional high-intensity bridge workloads until one of:

1. oMLX upstream releases a fix wrapping `check_error` in `try/catch`, OR
2. We write a local-mcp-toolbelt-side circuit breaker that detects oMLX
   abort + auto-retries the request after launchd restart (workaround, not
   solution), OR
3. We accept the abort risk and rerun eval with extra-conservative
   settings (single tier per session, hot_cache 4GB)

Phase 1 eval and 3rd PA Review candidate (copilot MCP scan) are deferred
until option 1 or 2 lands.

## Update — HEAD upgrade resolves the stress-test crash (2026-05-11)

Upgraded oMLX from stable 0.3.8 → HEAD-3d62ea0 (containing #1126 OOM
guard + #1146 async_eval cache-store + #1101 hot-cache shutdown flush).
Build from source on `brew install --HEAD jundot/omlx/omlx` took 2m 08s.

Smoke-tested with the exact stress pattern that previously aborted:

| Call | Wall | Notes |
|---|---|---|
| 14B cold | 8 s | hot_cache=6GB cold-load |
| 14B warm-1 | 3 s | cache hit |
| 14B warm-2 | 2 s | cache hit |
| 4B tier-swap | 3 s | Tier B cold |
| 14B after swap | 2 s | **the pattern that previously SIGABORTED at 17s uptime** |

oMLX still alive after the run; zero new `Python-*.ips` reports in
`~/Library/Logs/DiagnosticReports/` since restart. Crash mode does NOT
reproduce on HEAD-3d62ea0 with hot_cache=6GB.

**Verdict**: option 1 (upstream fix) is effectively satisfied via the
HEAD upgrade. Phase 1 eval is unblocked. The Phase 2 circuit-breaker
patch in MlxHttpBackend is still worth shipping as cheap insurance
against future variants of this abort surface (`check_error` is still
unguarded in HEAD; upstream may regress).

## What can safely continue

- Documentation work (PA Review for v0.6.0 already has 2/3 candidates)
- `gem` / `copilot` / `nv_pro` calls — they don't touch oMLX
- Bridge calls below ~2K tokens input on Tier B — well within Metal headroom
- Single bridge call at a time (no concurrent fan-out)

## What's next
