  all 3 tier runs (logs at `.claude/diagnostics/eval-tier-{B,C,D}.log`).
- Conclusion: HEAD-3d62ea0 + 6 GB hot_cache + the eval workload pattern
  hit the abort surface in maybe 1-in-N attempts, where N ≥ 27 today.
  The earlier Stage-1 transform crash was likely race-conditioned;
  follow-up runs of the identical workload do not reproduce it.

## What this validates

1. **Tier B is the workhorse**. Every tool routes through 4B-Instruct
   today; latency is bounded at ~20 s for the largest task (chunked
   summarize). The 60 s wall is not a constraint at this scale.
2. **Tier C (8B) is fine for `summarize-long` / chunked**. Worst case
   in this run was 29.8 s — comfortable headroom under the 60 s wall.
3. **Tier D (14B) is viable for opt-in classify + transform** with
   hot_cache warm. Cold-load tax is 14-20 s (first call), warm is
   6-7 s for classify and ~21 s for transform. All under 60 s.

## What this does NOT prove

- **Long-input transform / extract on 14B** was not exercised here.
  Real-world payloads with 5-10 K-token prefill on 14B would push
  toward the 60 s wall.
- **Crash MTBF** under this load is at most "≥ 27 trials between
  crashes" — that's a lower bound, not an upper bound. The bug is
  still upstream-unfixed; we just got lucky this run.

## Interpretation

### What this changes about prior claims

The `docs/scope-memos/v0.5.0-shipped-2026-05-10.md` memo describes
v0.5.0 as "shipped". It was never quality-validated against the
eval rubric — this run is the first real data we have. The scope
memo should be updated to mark §"Validation" as **not done in
v0.5.0; first attempt in v0.5.1 evaluation 2026-05-11**.

### What this does NOT change

- Bridge is still useful when frontier just needs *compression* of
  external content (the original "save Claude tokens" goal) — the
  local model's output doesn't have to match Opus-grade gold for
  that to be worth doing.
- The 60 s wall analysis stands — none of the 27 trials approached
  the wall, and v0.6.0's filesystem-return-channel is still the
  right structural move because it future-proofs against larger
  workloads, not because today's workloads need it.

### Judge model caveat

NIM's `qwen3-coder-480b-a35b-instruct` is a **coder-tuned** model.
Coder bias against prose summaries is a known pattern. A re-run
with a prose-fluent judge (e.g. Claude Sonnet via Anthropic API, or
GPT-5 via OpenAI) would likely produce different absolute scores —
but the **relative ordering between tiers** (B ~ D < C) is the
signal worth preserving for now.

### Tier ordering surprise: 14B is NOT obviously better than 4B

- 4B-Instruct mean: 2.54
- 14B mean: 2.50

The difference is within judge noise. If this holds under a non-coder
judge, **Tier D's opt-in cost (longer latency, larger memory, more
crash risk under the unguarded `check_error` upstream bug) is not
paying for itself on the current tool mix**. Candidate v0.6.0
decision: drop Tier D as a default promotion target; keep it only
for tasks where a real quality lift is empirically demonstrated.
