# v0.5.0 process failures — retrospective (2026-05-07)

Five concrete process failures during the v0.5.0 Tier-D session. Each one
cost wall-clock or frontier tokens; each has a corrective rule.

## 1. Built `mlx-bridge-server.py` without Prior Art Review

**What happened**: Wrote a 238-line custom FastAPI wrapper around `mlx_lm` to
serve `/v1/chat/completions`. The Auditor's feature-intake-rule (≥3 candidates
in `docs/prior-art/` before any build) was skipped.

**Cost**: ~2 hours of build + the entire R1/R2 multi-AI review needed to
discover the file should be deleted. Ended up replaced by oMLX
(`jundot/omlx`) — exactly the kind of result a 30-minute PA-review would have
surfaced.

**Evidence**: git log shows `mlx-bridge-server.py` created and deleted in
the same v0.5.0 cycle. Three production-ready alternatives existed at decision
time (`mlx-omni-server`, `mlx_lm.server`, `oMLX`) — none were checked.

**Corrective rule**: any new file under 100 lines that wraps an external
inference engine is a PA-review trigger. No exceptions.

## 2. Misidentified oMLX project (`mlx-omni-server` vs `jundot/omlx`)

**What happened**: When reviewers cited "oMLX" I assumed they meant
`madroidmaq/mlx-omni-server` (713 ⭐, MIT). Wrong — they meant
`jundot/omlx` (12.5K ⭐, Apache-2.0, Claude-Code-tuned). Two completely
different projects with similar names.

**Cost**: One full R2 review pass framed against the wrong project.
~1500 frontier tokens of incorrect analysis.

**Evidence**: my R1 critique said "oMLX is just OpenAI-compat shim, no
Claude Code integration" — that's true of mlx-omni-server, false of
jundot/omlx (which has explicit `claude_code` settings section with
opus/sonnet/haiku slots and SSD KV cache).

**Corrective rule**: when a name is ambiguous, fetch the canonical URL
before any analysis. `gh search repos <name>` first, conclusions second.

## 3. Streaming critique — wrong applicability lens

**What happened**: included "no streaming" as a problem with the MLX path
in R1 review. The user pushed back: "MLX 是用来调用本地模型的，这个
token-level 有什么用？"

**Cost**: ~600 tokens of misdirected critique.

**Evidence**: MCP tools return single complete results. Streaming exists
inside the model serving stack but provides zero client-side value for
tool-completion semantics. This is obvious in retrospect; it required the
user to point it out.

**Corrective rule**: before listing a "missing feature" as a flaw, ask
"would the consumer of this API observe the difference?" If the answer
is no, drop the critique.

## 4. Bridge-usage failures (×6 in one session)

**What happened**: Read R1/R2 reviewer outputs (~5-7KB each) and the
oMLX playbook (8.4KB) directly into frontier via `awk`/`head` instead of
through `mcp__ollama-bridge__extract` with `source_uri=file:///tmp/*.md`.

**Cost**: Estimated 25-40K frontier tokens of raw doc content that should
have entered as bridge-summarized output.

**Evidence**: When finally fixed (R2 extract on r2-copilot.md): single
bridge call returned a 800-token structured summary instead of the
8KB raw text. Net savings: ~1845 tokens for that one document.

**Corrective rule**: any tool result returning a path on `/tmp/` or any
file > 1KB triggers an automatic bridge call before any direct read.
This belongs in `~/.claude/.../bridge-trigger-checklist.md` (already
exists; not enforced).

## 5. Reverse-engineered API instead of reading docs

**What happened**: when oMLX was first installed, used `curl` against
`/v1/models`, `/v1/chat/completions`, `/health`, etc. to discover the API
shape. Should have used `copilot` (which has GitHub integration) to fetch
the official `jundot/omlx` README + docs first.

**Cost**: ~30 minutes of trial-and-error curl + ~3000 frontier tokens of
JSON parsing of unknown responses.

**Evidence**: when finally read via `copilot --prompt 'fetch jundot/omlx
docs and extract the model setup playbook'`, got a complete answer
including the bundled-Python download path in 90 seconds.

**Corrective rule**: a new external dependency means **read the docs
first** (via `copilot` or bridge-fetched URL) BEFORE any curl-based
exploration. Documented behaviour beats inferred behaviour.

The bridge-non-use cost dominates by an order of magnitude. The
project's *whole stated purpose* is bridge-mediated token saving for
Claude Code; failing at that during the session that built v0.5.0 is
the rule violation that matters most.

## Process changes for v0.6.0

1. **Pre-flight checklist on session start**: before reading any
   `/tmp/` file > 1 KB, route via bridge `extract`/`summarize-long` with
   `source_uri`. Hard rule, no exceptions.
2. **PA-review trigger rule**: any new file < 100 lines wrapping an
   external service requires `docs/prior-art/<name>.md` with ≥3
   candidates. Auditor (user) blocks merge if missing.
3. **Doc-fetch before curl**: a new external dep MUST start with one
   `copilot` or bridge-summarized doc fetch. The README is the spec;
   curl is the verification.
4. **Name disambiguation**: when a project name appears in another
   reviewer's output, fetch the GitHub URL via `gh search repos
   <name>` before quoting or critiquing.
