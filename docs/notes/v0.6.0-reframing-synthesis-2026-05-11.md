## Final architecture (R2 consensus)

```
NEW TOOLS:

enqueue_job(tool: string, args: object, opts?: { thinking?: 'on'|'off'|'auto' })
  → {
      job_id: string,
      result_uri: string,      // file:// path where result will appear
      est_ms: number,          // server's estimate
      wait_command?: string    // present iff client.capabilities includes 'bash'
    }

check_progress(job_id: string)
  → { status: 'queued'|'running'|'done'|'error',
      progress?: number,        // 0-100 if known
      eta_s?: number,           // remaining
      error?: string }

read_job_result(job_id: string)
  → { content: string }            // inline, preferred (works for sandboxed clients)
  | { file_path: string }          // fallback for very large results
  | { error: string }

MODIFIED TOOLS (all existing):

All current tools (summarize, classify, extract, transform, etc.) gain:
  thinking?: 'on' | 'off' | 'auto'   // default = 'auto'

  'auto' resolves to a server-side per-tool registry:
    classify, extract, transform, diff-semantic-index → 'on'
    summarize, summarize-long, summarize-long-chunked → 'off'

  Override path:
    - Per-call: pass thinking explicitly
    - Per-tool env var: OMCP_THINKING_<TOOL>=on|off (e.g. OMCP_THINKING_SUMMARIZE)
    - Global env var: OMCP_THINKING_MODE=on|off (overrides registry)

CROSS-CLIENT BEHAVIOR:

- All clients (Claude Code, Desktop, Cursor, Cline, etc.) can use the
  enqueue_job → check_progress → read_job_result triad as a portable
  long-running-call primitive.
- Claude Code can additionally use wait_command via its built-in Bash
  tool (10min cap) to consolidate poll+read into a single Bash invocation
  — saves MCP round-trips for high-latency jobs.
- Sandboxed clients (Cursor in some configs) may not have file:// read
  access; for them read_job_result returns content inline.
- Capability advertisement: enqueue_job returns wait_command only when
  the client has advertised `bash` capability (or env var
  OMCP_ASSUME_BASH_CLIENT=1).

EXISTING SYNCHRONOUS TOOLS:

All current tools remain synchronous by default. enqueue_job is the
EXPLICIT async path for tasks the caller expects to exceed their
client's request timeout. Phase 1 data shows most calls fit sync (60s
wall) even on Tier D; async is the opt-in escape hatch.

CIRCUIT BREAKER:

The MlxHttpBackend circuit-breaker (commits 839fbac, 09783e8, 994f01c)
remains in place. It masks oMLX SIGABRT crashes within both sync and
async paths transparently.
```

## Cut-if-pressured: bash-wait optimization

3/4 R2 voices vote cut bash-wait first. Reasoning: universal polling
still works for Claude Code; you just pay an extra MCP roundtrip every
few seconds during wait. The optimization is real (saves overhead) but
not critical to the wall-bypass story.

## Open questions for scope memo

1. **`'auto'` thinking semantics** — is it a literal alias for the
   per-tool registry (deterministic, gem/copilot/nv_pro position), or
   can it eventually evolve into a model-driven decision (deferred per
   nv_pro R2)? Default for v0.6.0: alias only.

2. **Capability detection** — how does the bridge know whether the
   client has a Bash tool? Options:
   - MCP capability handshake (clean but requires client cooperation)
   - User-config env var `OMCP_CLIENT=claude-code`
   - Always return `wait_command` and let clients ignore it

3. **`progress` field on `check_progress`** — when do we actually have
   progress data? For chunked summarize, yes (chunks_done / total). For
   monolithic classify/transform, no. Should be optional and only
   populated where meaningful.

4. **Tier D status given Phase 1 data** — 14B mean 2.50 vs 4B 2.54 in
   no-think mode. Per nv_pro's prior analysis, Tier D may not earn its
   keep on current tool mix. v0.6.0 should at minimum demote Tier D to
   experimental until thinking-on eval data establishes a real lift.

5. **Telemetry on circuit-breaker frequency** — already shipped (commit
   994f01c). v0.6.0 should add a `/health` extension or `_meta` field
   surfacing breaker-retry-rate so we can monitor oMLX stability over
   time without grepping stderr logs.

## Risks (R2 verified)

| Risk | Source | Mitigation |
|---|---|---|
| Sandboxed clients can't read file:// paths | gem R2 | `read_job_result` returns inline content as primary; file_path as fallback for large results |
| Bash-wait coupled to Anthropic's tool semantics | gem + copilot R2 | Treat as opt-in via capability detection; never required for correctness |
| `auto` thinking flag ambiguity | claude R2 | Document `'auto'` as deterministic alias for per-tool registry; no LLM-decides path in v0.6.0 |
| oMLX SIGABRT still unguarded upstream | session data | Circuit-breaker masks; upstream report draft ready |
| Thinking on prose hurts quality | thinking eval data | Per-tool default `off` for prose tools |
