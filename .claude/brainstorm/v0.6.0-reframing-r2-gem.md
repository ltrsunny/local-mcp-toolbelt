### 3. CROSS-CLIENT POLICY (≤100 words)

The minimal cross-client design for Claude Code, Desktop, and Cursor must be API-based polling. Bash-wait is exclusive to Claude Code. A filesystem-return-channel is brittle, assuming client-side filesystem access that may not exist in sandboxed environments like Cursor.

The only viable, universal pattern is `enqueue_job` returning a `job_id`, followed by client-side polling of a `check_progress(job_id)` endpoint. This is a standard, robust async pattern that requires no special client capabilities beyond making HTTP requests, ensuring the bridge works identically across all three platforms.

### 4. ONE FINAL ARCHITECTURE (≤150 words)

The v0.6.0 architecture will be a synthesis of A+C: an async job system with a filesystem result and a per-tool thinking flag.

*   **API Surface:**
    1.  `enqueue_job(tool: str, args: dict, thinking: str = 'auto') -> {job_id}`
    2.  `check_progress(job_id: str) -> {status: 'running'|'error'|'done', result_path: str|None}`
*   **Behavior:** The client calls `enqueue_job`, then polls `check_progress`. When `status` is `done`, the `result_path` is provided, and the client reads the output from that file. `thinking: 'auto'` uses server-side defaults (on for logic, off for prose).
*   **Cross-Client:** This polling mechanism is universal.
*   **Engineering Cost:** **Medium (M)**. Requires building the job queue, status endpoint, and file-based result store.
*   **Cut if forced:** The `thinking` parameter. The core goal is solving the 60s wall; quality tuning is secondary.
