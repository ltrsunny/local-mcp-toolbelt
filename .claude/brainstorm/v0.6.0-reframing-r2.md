## 1. Concrete v0.6.0 architecture

**Synthesis E (B + C + partial D)** — keep MCP minimal, weaponize Bash, expose thinking per-tool.

API surface:
```

  → { job_id, result_path, wait_command, est_ms }

# wait_command is a ready-to-paste bash one-liner:
#   "while [ ! -f /path/to/<id>.json ]; do sleep 5; done; cat /path/to/<id>.json"
# Caller (Claude) passes this to its own Bash tool — 10min cap >> any MLX task.

read_job_result(job_id) → result          # MCP-side path, instant once file exists
list_jobs() → [{job_id, status, age_s}]   # optional; small surface
```

Per-tool `thinking` flag is the API for the v0.5.x → v0.6.0 quality lift.
Default per-tool (set in `server.ts` registry, not passed by caller):
- `classify`, `extract`, `transform`, `diff-semantic-index`: **`on`** (reasoning helps)
- `summarize`, `summarize-long`, `summarize-long-chunked`: **`off`** (prose loses 0.5 in eval data)

Trade-off: dual-path (MCP for short / structured-output / cached results; Bash for long thinking-on runs). Caller picks via `wait_command` vs `read_job_result`.

Main risk: Bash tool's 10min cap is an Anthropic product decision; could shrink. Mitigation: monitor; fall back to in-MCP `check_progress` polling if it ever does.

Cost: **S–M**. v0.3.0 job worker is reusable. New: `wait_command` string generation (~5 LOC), per-tool thinking default registry (~30 LOC), env-var override (`OMCP_THINKING_<TOOL>=on|off`).

## 2. Challenge B (bash-wait): brittle or sound?

**Structurally sound, not brittle.** Anthropic ships `Bash` with an explicit 10min cap *because* shell ops are expected to run long (builds, tests, long-poll). Removing that would break vastly more than our use case. The brittleness, if any, is shell-portability of `while [ ! -f ]; do sleep 5; done; cat` — use POSIX-only subset, no `[[` or zsh-isms, and it's safe across `/bin/sh`, `bash`, `zsh`.

## 3. Challenge thinking-everywhere

**Signal, not noise.** Direction is consistent with Qwen3 model card behavior: reasoning models on prose tasks tend to over-generate and split structure, hurting summarization scores. Sample is small but two tasks moved the same way (summarize-long −1.0, classify −0.7 on judgable rows). v0.6.0 should expose per-tool defaults; auto-mode (let model decide) requires more eval data and stays out of scope.

## 4. Prediction

Final v0.6.0 will be **synthesis B + C** (bash-wait + per-tool thinking). A (filesystem + `check_progress`) gets dropped or reduced to a thin fallback — `wait_command` makes a dedicated progress MCP tool redundant for the long-path. The cleaner architecture won't have both.

### 1. Proposed v0.6.0 Architecture: Option E (Hybrid Async + Per-Tool Thinking)
**Architecture:** Implement a hybrid model combining **Bash-wait for long-running jobs** and **per-tool thinking flags** for quality control.
*   **API Surface:**
    *   `run_async_tool(name, args)`: Returns an immediate job ID. Claude Code invokes a companion tool `wait_and_fetch(job_id)` which executes a shell loop (`while [ ! -f done ]; do sleep 2; done; cat result`) using the built-in Bash tool (Fact 2). This bypasses the 60s MCP client hardcode (Fact 1) by offloading the wait time to the Bash tool's 10-minute window.
    *   `summarize`, `classify`, `transform` tools gain a `thinking: "on" | "off" | "auto"` parameter. Default is `"auto"` (off for prose, on for logic). This leverages `OMCP_THINKING_MODE=on` to utilize `reasoning_content` without burning `MAX_OUTPUT_TOKENS` (Fact 6).
*   **Key Trade-off:** Increases API complexity by splitting execution paths (sync vs. async) but guarantees stability against the 60s wall while optimizing token usage.
*   **Main Risk:** Race conditions in the filesystem polling mechanism if `wait_and_fetch` isn't atomic; reliance on the Bash tool remaining stable.
*   **Engineering Cost:** **Medium**. Requires implementing the job queue filesystem logic and refactoring tool handlers to accept the `thinking` enum, building on the existing circuit-breaker stability (commits 839fbac, 09783e8, 994f01c).

### 2. Challenge B (Bash-wait): Brittle or Sound?
The "Bash-wait" pattern is **structurally sound** because it aligns with Anthropic's explicit design intent: the Bash tool provides a 10-minute timeout specifically for long-running shell operations (Fact 2), effectively decoupling the wait time from the MCP client's 60s hardcode (Fact 1). While theoretically brittle if Anthropic removes the Bash tool or reduces its timeout, it is currently the most robust native workaround available, superior to complex polling mechanisms that consume MCP call quotas.

### 3. Challenge Thinking-Everywhere: Noise or Signal?
The mixed eval data (transform +1.0, summarize −0.5) is a **clear signal** that reasoning traces degrade performance on prose-heavy tasks (likely due to over-analysis or style drift) while significantly boosting logical transformation accuracy. Consequently, v0.6.0 must **reject "thinking-everywhere"** in favor of the proposed per-tool `thinking` flag, defaulting to `off` for summarization/classification and `on` for transformation/chunking to maximize quality without unnecessary latency or quality penalties.

### 4. ONE Prediction
v0.6.0 will ship as a synthesis of **Option B and C**, utilizing the Bash tool's 10-minute window to bypass the 60s MCP wall for async jobs while gating `OMCP_THINKING_MODE` behind explicit per-tool parameters to address the quality divergence seen in Tier C/D trials.
