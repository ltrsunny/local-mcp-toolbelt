I propose **Option F: Context-Aware Gating via Edit-Intent Heuristics**.

Given the PM's history of rejecting soft-rule slippage, Option B (exemption) is politically untenable, while Option A (blanket revert) imposes unacceptable friction on the primary workflow: iterative editing. The core tension lies in the hook's inability to distinguish "Read-for-Edit" from "Read-for-Understanding." Solving this via user-side discipline (Option C) fails the PM's requirement for automated enforcement, effectively reintroducing the very inconsistency the hook system was designed to eliminate.

The proposed solution modifies the hook to inspect the *calling context* rather than just the file path. If the `Read` request originates immediately preceding an `Edit` operation on the same file (detectable via conversation state or tool-call sequencing), the hook allows direct access. This treats the "Read-then-Edit" pattern as an atomic transaction, exempting it from the bridge requirement. Conversely, isolated reads of scope memos remain gated to the bridge, enforcing the "large input → small structured output" principle for analysis tasks.

The realistic friction cost is negligible for editing workflows, as the atomic transaction remains uninterrupted. For analysis tasks, the friction is intentional and necessary, aligning with the project's token-saving goals. This approach avoids the complexity of maintaining per-session environment variables while satisfying the PM's demand for rigorous, automated governance.

The "for-Edit vs for-understanding" distinction is too critical to leave to user discipline. Relying on developers to manually toggle modes or remember exemptions invites the exact drift that caused the initial design failure. By encoding this logic into the hook's decision matrix, we preserve the system's integrity without sacrificing the fluidity required for source-tier interactions. This maintains the architectural discipline of Sub-pattern A while acknowledging the unique requirements of document mutation.
