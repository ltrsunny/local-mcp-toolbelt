PROBLEM: local-mcp-toolbelt (MCP server, Apache-2.0, this project) hits a hard
60-second wall-clock on every MCP request from Claude Code. Claude Code (the
MCP client, closed source) has this timeout HARDCODED — cannot be raised via
settings.json, any env var (MCP_TIMEOUT controls only startup), or any docs.
See anthropics/claude-code#5221, #22542.

v0.3.0 added an async-job pattern (enqueue-job → wait_for_job → read_job_result)
where wait_for_job long-polls up to 45s and Claude re-calls if not done. This
works but:
  (a) wait_for_job itself has the same 60s wall, so each poll caps at 45s
  (b) Claude must manage the retry loop manually (cognitive cost + tool calls)
  (c) for any single tool call genuinely needing >60s of model time (long
      Tier C summarize on 25K-word docs, 14B classify on big input), the
      job runs in background but each wait_for_job hop costs an MCP request

CONSTRAINTS:
- Cannot modify Claude Code (closed source). Cannot rely on MCP server push
  / SSE — MCP transport is request-response.
- Apple Silicon 16GB Mac. oMLX serializes generation on Metal (one model
  generating at a time).
- Per-call work can legitimately be 30-300s. Bridge tools: summarize,
  summarize-long, summarize-long-chunked, classify, extract, transform,
  diff-semantic-index, plus the v0.3.0 async-job triad.
- Solutions should work TODAY with current MCP spec (not a future spec
  proposal). Other MCP clients (Claude Desktop 240s, Cursor configurable)
  are out of scope — we're optimizing for Claude Code.
