### 1. Pick ONE concrete v0.6.0 architecture

**Architecture:** Semantic Thinking API + Circuit-Breaker Reliance (Synthesis of C & D)

**API Surface:**
All core MCP tools (`summarize`, `classify`, `transform`) receive a new parameter:
```json
{
  "thinking": {
    "type": "string",
    "enum": ["auto", "on", "off"],
    "description": "Control reasoning trace generation. Default: 'auto'."
  }
}
```
The server evaluates this flag and passes the intent to `MlxHttpBackend`, dynamically applying the vestigial `/no_think` suffix or enabling `OMCP_THINKING_MODE=on` (Fact 6).

**Key Trade-off:**
This prioritizes immediate output quality (currently failing at ~2.5/5) over engineering an elaborate async workaround for a 60s wall that none of the 35 eval trials actually hit (Facts 3 & 4). We absorb the ~5s oMLX crash-retry latency tax (Fact 5) directly within the existing 60s synchronous window.

**Main Risk:**
If users supply significantly larger contexts, or if a heavier model pushes base inference latency to ~50s, a single unguarded `mlx::core::gpu::check_error` abort and subsequent circuit-breaker retry will push the total request time over Claude Code’s baked-in 60s limit (Fact 1), resulting in a sudden client-side failure.

**Engineering Cost:** Small (S).

### 2. Challenge B (bash-wait): brittle or sound?

The Bash-wait pattern is structurally brittle. It couples the bridge's core async architecture to the proprietary behavior of one specific client's built-in tool (Claude Code's Bash environment) rather than adhering to standard MCP primitives. If Anthropic restricts the Bash tool's 10-minute timeout or alters how it handles polling loops in a future update, the bridge's async capabilities will instantly break.

### 3. Challenge thinking-everywhere

The mixed eval data is a clear signal, not noise: deep reasoning models frequently over-complicate simple text summarization or drift from strict formatting constraints, directly causing the `03 classify` parse failure. Because thinking explicitly hurts rigid classification but helps complex refactoring (`05 transform` UP 1.0), v0.6.0 must expose a per-tool `thinking` parameter, allowing Claude to contextually disable reasoning for schema-bound tasks.

### 4. ONE prediction

I predict v0.6.0 will ultimately ship a synthesis of **Option C (Per-tool thinking) and Option A (Filesystem-return-channel)**, because as users leverage the new thinking API on larger tasks, the compounding ~5s delays from oMLX circuit-breaker retries will inevitably push real-world workloads past the 60s wall.
