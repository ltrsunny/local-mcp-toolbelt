# R2 — Claude (claude-opus-4-7)

## 1. Settle Bash-wait

Partial concede. My R1 declared Bash-wait "structurally sound" but missed cross-client implications. gem's strongest argument: **"couples the bridge's core async architecture to the proprietary behavior of one specific client's built-in tool"**. That's correct as a category statement — Bridge is explicitly cross-client (CLAUDE.md names Cursor, Cline, OpenClaw, Zed as targets).

New position: Bash-wait is **sound for Claude Code specifically** (Anthropic won't shrink Bash 10min — would break their own product) but **insufficient as the only long-job primitive** for a cross-client bridge. Solution: ship both. They share 90% infrastructure (job queue, file storage); only the caller-side wait mechanism differs.

The argument I must defeat (gem's): "future Claude Code update could break our reliance" — mitigated by treating Bash-wait as opt-in `wait_command` field that clients can ignore, not as the only return path.

## 2. Settle thinking default

3/4 voted `auto`, but reading carefully: nv_pro defined "auto" as **"off for prose, on for logic"** — that's exactly my per-tool position with a different label. copilot said "with heuristics" but didn't name them. gem said "auto" without specifying the rule.

**Actual vote: 3-of-4 (claude + nv_pro + copilot)** for **server-side per-tool table**; 1-of-4 (gem) for unspecified "auto". Settle: `thinking` field accepts `'on'|'off'|'default'`. `'default'` (the default value, no pun) consults a per-tool registry in `server.ts`:
- `classify`, `extract`, `transform`, `diff-semantic-index`: `'on'`
- `summarize`, `summarize-long`, `summarize-long-chunked`: `'off'`
Caller can always override.

## 3. Cross-client policy

The simplest design: **most clients don't need async**.
- Claude Desktop: 240s timeout — every Phase 1 trial fits sync
- Cursor: configurable, defaults generous — fits sync
- Cline / OpenClaw / Zed: configurable
- **Claude Code: 60s — this is the only client where async matters**

So `enqueue-job` + `wait_command` is **Claude-Code-specific optimization, not cross-client requirement**. Other clients call the underlying tools (`summarize`, `transform`, …) synchronously. `enqueue-job` is opt-in. Filesystem-return-channel as a separate cross-client primitive (copilot's position) is over-engineering for v0.6.0 — defer until a non-Claude-Code client actually hits its wall.

## 4. ONE FINAL ARCHITECTURE

```
Added:
  enqueue-job(tool, args, opts?: { thinking?: 'on'|'off'|'default' })
    → { job_id, result_path, wait_command, est_ms }
  read_job_result(job_id) → result body

Modified all existing tools:
  + thinking?: 'on'|'off'|'default'  (default = 'default' → per-tool table lookup)
  Registry (server.ts): classify/extract/transform/diff-semantic-index → 'on',
                        summarize*/transform-prose → 'off'

Cross-client:
  - Sync tools work everywhere unchanged
  - enqueue-job opt-in (Claude Code uses it; other clients ignore it)
  - wait_command is Bash one-liner; clients without Bash poll read_job_result themselves

Cost: S-M (~5 LOC wait_command gen + 30 LOC per-tool table + reuse v0.3.0 queue + env var override OMCP_THINKING_<TOOL>=on|off)

If scope forced: drop the per-tool registry, ship thinking flag with no default
(caller specifies every time). Loses ergonomics but ships smaller.
```

## 5. Steelman of my R1 being wrong

I declared "Bash-wait structurally sound" using only Claude-Code-as-reference. The bridge's CLAUDE.md spec lists 5+ MCP clients. A primary path that only works for one client breaks the cross-client promise — that's a real product-correctness bug, not just an aesthetic preference.
