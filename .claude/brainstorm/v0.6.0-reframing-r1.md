## Context (6 hard facts from 2026-05-11)

1. **60s wall is Claude Code CLIENT hardcode, not MCP protocol or LLM
   limit.** MCP_TIMEOUT env var only controls startup; per-request timeout
   is baked in. anthropics/claude-code#5221, #22542.

2. **Claude Code's Bash tool has 10-minute timeout** (independent of MCP,
   configurable per call up to 600000ms). It's a built-in tool, not a
   user-installable MCP server.

3. **Phase 1 eval (no-think mode, the current bridge default)**: all 3
   tiers REJECT at mean ~2.5/5. Latency + RAM are NOT the binding
   constraint — none of 27 trials approached 60s wall.

4. **Thinking-mode subset eval (Tier C+D, 8 trials)**: mixed signal:
     - C 01 summarize-long: 3.0 → 2.0 (DOWN 1.0)
     - C 02 chunked:        2.5 → 3.0 (UP 0.5)
     - D 03 classify:       2.7 → 2.0 (DOWN; 1 parse failure)
     - D 05 transform:      2.0 → 3.0 (UP 1.0)
   Wall times all under 60s, so the "thinking needs wall fix" hypothesis
   from initial framing is partially wrong — thinking already fits the
   wall today; it's the QUALITY signal that's mixed.

5. **oMLX still has unguarded `mlx::core::gpu::check_error` abort surface
   even on HEAD-3d62ea0**. Our `MlxHttpBackend` circuit-breaker
   (commits 839fbac, 09783e8, 994f01c) masks the crash → restart →
   retry behind a ~5s latency tax.

6. **oMLX exposes `reasoning_content` separately from `content`** —
   thinking trace doesn't burn our MAX_OUTPUT_TOKENS cap. The
   historical `/no_think` suffix is a vestigial workaround from the
   Ollama/llama.cpp era. Setting `OMCP_THINKING_MODE=on` removes it.

  is enough for v0.5.x; v0.6.0 only needs better quality, not wall fix.
- **E. Combination of the above.**
