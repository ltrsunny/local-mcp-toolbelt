### 1. SETTLE BASH-WAIT
**Verdict:** Adopt a **Dual-Path Strategy** (Bash-wait primary, Filesystem-fallback).
The "Sound" camp (**claude**, **nv_pro**) correctly identifies that the Bash tool's 10-minute cap is an explicit design feature for long ops, making it the optimal primary path for Claude Code users today. However, the "Brittle" camp (**gem**, **copilot**) raises a valid architectural risk: coupling core async logic to a proprietary client tool creates a single point of failure if Anthropic alters sandboxing.
**Resolution:** Implement Bash-wait as the default for Claude Code to leverage immediate stability, but wrap the filesystem return logic (Option A) as a mandatory fallback triggered by environment variable or client capability detection. This satisfies **claude**'s efficiency needs while addressing **gem**'s concern about proprietary lock-in. We do not choose one; we layer them.

### 2. SETTLE THINKING DEFAULT
**Verdict:** Explicit Per-Tool Defaults (Reject `auto`).
While 3/4 peers (**gem**, **copilot**, **nv_pro**) voted `auto`, the term is ill-defined without a robust heuristic engine the server currently lacks. As **gem** notes, reasoning actively harms rigid schema tasks (classify) but helps logic (transform). An `auto` flag delegates this binary decision to the LLM at runtime, introducing latency and inconsistency.
**Resolution:** Adopt **claude**'s position: hardcode per-tool defaults in the server registry (`transform: on`, `summarize: off`). This guarantees the eval-backed quality lift immediately. `auto` can be a future enhancement once we have telemetry to define the decision boundary, but v0.6.0 requires deterministic behavior based on the known divergence in task types.

### 3. CROSS-CLIENT POLICY
**Verdict:** Minimal Viable Cross-Client = `check_progress` + Filesystem Return.
Bash-wait is exclusive to Claude Code. For Claude Desktop, Cursor, and Cline, the bridge must rely on standard MCP roundtrips.
**Design:**
1.  **Unified Entry:** `enqueue_job` always returns a `job_id`.
2.  **Polling Standard:** All clients use `check_progress` (MCP native) to poll status.
3.  **Result Retrieval:** Once done, results are fetched via `read_job_result` (reading the filesystem artifact).
This avoids engineering two distinct async mechanisms. Bash-wait becomes a "local optimization" for Claude Code users who can inject the specific wait command, but the core architecture relies on the universal polling pattern.

### 4. ONE FINAL ARCHITECTURE
**v0.6.0 Final Spec:**
*   **Tools:** `enqueue_job(tool, args, thinking?: 'on'|'off')`, `check_progress(job_id)`, `read_result(job_id)`.
*   **Defaults:** `thinking` defaults hardcoded per tool (e.g., `transform`→`on`, `summarize`→`off`). No `auto`.
*   **Async Mechanism:** Primary path is filesystem-based artifacts polled via `check_progress`.
*   **Claude Code Optimization:** If client detects `claude-code`, suggest the Bash one-liner for `check_progress` to bypass MCP timeout limits, but fallback to standard polling if Bash fails.
*   **Engineering Cost:** **Medium**. Requires job queue persistence and state management.
*   **Cut if Scope Constrained:** Drop the Bash-wait optimization entirely. Ship only the universal filesystem polling path. It is slower for Claude Code users but guarantees cross-client compatibility and reduces surface area.

### 5. ONE STEELMAN OF YOUR PRIOR R1 BEING WRONG
My R1 over-indexed on Bash-wait as a permanent solution. **gem** is right: relying on a specific client's shell implementation for core async architecture is a hidden dependency that could break silently, making the "universal" polling path the only safe long-term bet.
