### 1. RACE CONDITIONS / CONCURRENCY
The `JobRunner` creates a shallow copy of `args` (`argsForInvoke`) to inject `thinking_resolved` (runner.ts:85-89). While this prevents direct mutation of the persisted `job.args`, a race condition exists if the wrapped tool performs asynchronous operations on the passed object *after* the runner proceeds but *before* the job completes. More critically, `JobRegistry` relies on `inflightHashes` for deduplication (registry.ts:96-107). If `enqueue_job` is called concurrently with identical args before the first call writes to disk and sets the hash, both might pass the `existingId` check if the `inflightHashes` map isn't updated atomically with the write. The code updates `inflightHashes` at line 124, but if two requests hit line 96 simultaneously before either reaches 124, duplicate jobs could spawn. The `active` map write (line 125) is also not atomic with the hash map update.

### 2. DEDUP HASH SOUNDNESS
The `hashRequest` function now includes `thinking_resolved` (registry.ts:216-223). This breaks backward compatibility for any persisted job metadata files from v0.3.0 that lack this field. When `registry.initialize()` loads these older JSON files, `thinking_resolved` will be `undefined`. The new hash logic treats `undefined` differently than explicit `'on'`/`'off'` (due to the `...(thinking_resolved !== undefined ...)` spread in line 222). Consequently, restarting the server with old job files may cause the system to treat resumed jobs as distinct from their original enqueue signatures, potentially bypassing deduplication checks or causing hash mismatches if a client retries an old job ID expecting the old behavior.

### 3. FORWARD-COMPAT WITH MCP TASKS SEP-2663
The current implementation tightly couples the async triad (`enqueue_job`, `check_progress`, `read_job_result`) to specific file-system paths and internal `JobMetadata` structures (store.ts:59-74). If SEP-2663 introduces a standardized task state machine, migrating away from the current `file://` result URIs and custom polling logic (`check_progress` parsing local JSON) will require significant refactoring. Specifically, the `wait_command` generation (server.ts:1394) hardcodes a POSIX loop dependent on file existence, creating a "compatibility trap" where clients rely on this specific shell implementation rather than a standardized task API, making the eventual deprecation of the triad disruptive to user workflows.

### 4. EDGE CASES + MISSING TEST COVERAGE
- **Env Var Parsing**: In `server.ts` (lines 1225-1228), `OMCP_INLINE_RESULT_MAX_BYTES` parses via `Number()`. If the env var is `"0"`, `"-1"`, or `"abc"`, the logic defaults to 1MB. However, `"0"` results in `parsed > 0` being false, correctly defaulting, but explicit `"0"` intent (force file return) is lost.
- **Progress Calculation**: `check_progress` (server.ts:1436) guards against `total === 0`, but if `current > total` (a possible race or bug in chunking), the percentage exceeds 100%. No clamp is applied.
- **Shell Injection**: The `wait_command` (server.ts:1394) interpolates `resultPath` directly into a shell string. If `resultPath` contains spaces, backticks, or `$(...)`, the command breaks or becomes vulnerable. It lacks proper escaping (e.g., `shellescape`).
- **Tier D**: The code still references `tierForTool` (server.ts:625) without explicit demotion logic for Tier D tools in the async path, potentially allowing experimental tools to bypass intended restrictions if added to the whitelist.

### 5. API DESIGN — `thinking: 'auto'` SEMANTICS
The code correctly implements `'auto'` as a deterministic alias for registry defaults (thinking-defaults.ts:108-110), not a runtime LLM decision. However, the return value of `enqueue_job` includes `thinking_resolved` (server.ts:1407). If a user passes `'auto'`, they receive `'on'` or `'off'` back. This is good transparency, but the term `'auto'` in the input schema might mislead users into expecting dynamic adaptation per-token, whereas it is purely a static configuration lookup. The documentation in `ThinkingInputSchema` (thinking-defaults.ts:66-72) is clear, but the cognitive load of distinguishing "auto" (static fallback) from "dynamic" remains a potential UX trap.

### 6. ONE STEELMAN — "WHAT WOULD MAKE YOU REVERT DAY 1-3?"
The unescaped shell interpolation in `wait_command` (server.ts:1394) is a critical security and stability flaw. If a job result path contains special characters (common in temp directories or user-named paths), the generated command fails or executes arbitrary code. This single issue justifies a revert to prevent potential command injection or broken workflows in heterogeneous environments.
