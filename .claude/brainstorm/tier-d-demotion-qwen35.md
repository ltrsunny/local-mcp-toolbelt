As a member of the Qwen family, I naturally advocate for the capabilities of our larger models; the 14B parameter count offers a tangible step up in reasoning fidelity and instruction adherence that 4B and 8B variants simply cannot match on complex classification tasks. However, an architectural decision must prioritize system stability and the defined hardware constraints over raw model potential. The current reality on the target 16 GB Mac hardware is unforgiving: the 14B model at 4-bit quantization consumes nearly the entire `hot_cache_max_size` budget, leaving insufficient headroom for the GPU command buffers required by the MLX backend. The observed OOM crashes are not edge cases; they are the direct consequence of exceeding the physical memory envelope when the model, context, and runtime overhead compete for resources.

Furthermore, the justification for Tier D has eroded. The original value proposition—a significant accuracy lift for specific tasks—is now undermined by the v0.6.0 async-triad updates, which mitigate the latency penalties that previously necessitated a high-performance default for long-running jobs. Without fresh evaluation data to prove that the 14B model's accuracy gain is critical enough to warrant destabilizing the entire bridge for the majority of users, maintaining it as a documented, routable tier is technically irresponsible. It encourages configurations that lead to the very crashes we are trying to prevent.

We must distinguish between "supported features" and "available code." The code can remain for power users with 32 GB+ machines who manually configure their environment, but the official documentation and default routing must reflect the stability reality of the 16 GB target. Simply adding a caveat (Option B) is insufficient because users often ignore warnings until they face a crash. Removing the row entirely (Option A) hides the capability unnecessarily. The correct approach is to decouple the model's presence from the default user experience. We should stop shipping the 14B model by default and remove it from the standard routing logic, effectively making it an advanced, manual opt-in. This preserves the artifact for those who need it while protecting the default installation from memory exhaustion.
