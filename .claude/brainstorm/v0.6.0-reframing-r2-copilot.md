### 1. SETTLE BASH-WAIT (≤120 words)
Hybrid: ship the filesystem-return path as the canonical async channel and offer Bash-wait as an *opt-in* performance optimization only for Claude Code clients. claude and nv_pro correctly note Bash gives a pragmatic long-window workaround; gem and copilot correctly warn it’s brittle because it couples to a proprietary client. Mitigation: capability discovery (client advertises support), active telemetry, CI tests that exercise the wait loop, and an automatic fallback to MCP polling/check_progress if the Bash path fails or the wait window shortens. This keeps the win without risking a single-point-of-failure.

### 2. SETTLE THINKING DEFAULT (≤120 words)
“auto” must be explicit: server-side deterministic heuristic + per-tool registry fallback (claude’s position). Rule: if client passes thinking=on|off obey it. If thinking=auto, server uses:
1) tool registry default (registry: transform/extract/classify/diff → on; summarize* → off), then
2) override → on when request has structured/schema output (json_schema/output_schema) or keywords (refactor, prove, explain) or input tokens > 4k; else keep registry default. If heuristic is ambiguous, fall back to registry explicit default. This yields predictable, testable behavior and matches the 3:1 “auto” preference with claude’s safety.

### 3. CROSS-CLIENT POLICY (≤100 words)
Minimal cross-client design: implement filesystem-return-channel + check_progress as the universal MCP primitives (works on Claude Desktop, Cursor, Cline, etc.). Provide Bash-wait as an optional Claude Code optimization only. All clients must be able to discover supported capabilities (capabilities API). Engineering cost kept down by one canonical async path; Bash-wait is a small, documented add-on.

### 4. ONE FINAL ARCHITECTURE (≤150 words)
v0.6.0 (ship bet)
- API:
  - enqueue_job(tool:string, input:object, thinking: "on"|"off"|"auto" = "auto", output_channel: "file"|"mcp" = "file") → {job_id, result_uri, est_ms, capabilities}
  - check_progress(job_id) → {status, progress_pct?, eta_s?}
  - get_result(job_id) → {content:// | file:// | error}
  - optional: wait_command string returned when client capability `bash_wait=true`.
- Defaults: thinking="auto" (per above heuristic); output_channel="file".
- Cross-client: canonical = file+check_progress (S); bash_wait opt-in for Claude Code only.
- Engineering cost: Medium (M) — job queue + file plumbing + registry + telemetry.
- Cut if pressured: remove Bash-wait (keeps cross-client clean, cost S→M→S).

### 5. STEELMAN OF PRIOR R1 BEING WRONG (≤50 words)
Best counter: privileging Bash-wait made the bridge rely on a single-vendor primitive; if Anthropic tightens or removes the Bash semantics, the optimization would break consumers. A platform-agnostic file+polling design is safer and easier to maintain.
